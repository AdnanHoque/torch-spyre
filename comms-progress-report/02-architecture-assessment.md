# Architecture Assessment: The Comms-Collectives Contract

## Framing

The effort's north star — eliminate non-weight HBM spills in a Granite block by moving activations LX-to-LX — resolves, at the compiler level, into one question: **where does the line sit between the frontend (Torch/Inductor) and the backend (deeptools), and what crosses it.** The production scatter path answers that question for one corner of the 2-axis spill lattice. The two prototype branches probe the adjacent corners and, in doing so, stress-test the interface. The findings let me take firm positions rather than hedge.

## 1. "Frontend owns policy, backend owns realization" — RIGHT, and the branches confirm it

This split is correct, and the strongest evidence is that the branch that set out to *violate* it walked back to it. The stcdp-lane was framed as "Torch owns the physical plan," but its implemented state does **not** compute STCDP rows, piece sizes, or a movement schedule in Python. It records DLDSC coordinate metadata plus a class tag and flips a "don't reserve resident" routing hint; physical synthesis stays in the backend (`insertRelayoutSdsc → STCDPOpLx`). That is not a failure of the branch — it is the branch discovering that the backend already owns the information needed to realize the move (LX memory-tracker state, piece sizing via `getRelayoutPieceSize`, LX-vs-HBM fallback) and that mirroring it in Python would require keeping two copies of backend allocator logic in lockstep. The production path's own alternative-(a) analysis reaches the same conclusion independently: explicit frontend rows "hard-code a physical plan that depends on backend memory-tracker state Torch can't see."

So the split is right for a concrete, defensible reason: **realization depends on state that is structurally backend-local.** Policy — *should* this edge move on-chip, is it worth it, is it in scope — depends on graph-level knowledge (producer finality, single-writer, scope=computed-vs-weight) that is structurally frontend-local. The two knowledge sets don't overlap, so the split is not arbitrary; it tracks a real information boundary.

## 2. DLDSC coordinate metadata as the FE↔BE interface — RIGHT for pure movement, must be EXPANDED for the other two axes. The hypothesis holds.

**Confirmed for pure movement (scatter/broadcast/gather/all-gather): cardinality is implied, so the frontend should NOT hand-classify.** This is the sharpest position and the evidence backs it cleanly. The backend already derives the move from the two `coreIdToWkSlice_` maps — the mismatch *is* the plan. Cardinality falls out of the two distributions mechanically: count distinct producer slices vs consumer slices per dim and you have 1:1 vs 1:many vs many:1 vs many:many for free. The production notes flag exactly this ("the backend could classify cardinality straight from the coordinate maps, so the frontend's `communication_class` becomes a cross-check rather than a required input"). The deeptools verification corroborates: the single `insertRelayoutSdsc` path already builds STCDPOpLx pieces "including replicated output pieces" — i.e. the *same* coordinate-driven code covers scatter and multicast without a distinct lowering. **Therefore the stcdp-lane's hand-classification of `matmul_operand_broadcast` as a separate class is the wrong shape.** It should not be a frontend label; it should be a backend-derived property of the coordinate delta. The frontend's job on this axis collapses to *detection* (is there a mismatch worth moving?), not *classification*.

**Critically: did explicit planning prove DLDSC insufficient, or just un-generalized? UN-GENERALIZED, decisively.** The stcdp-lane's own wall is in the backend, not the interface. `runDcgForInputFetchNeighbor` is live but hard-coded to `DsTypes::INPUT` while the Granite value-side operand is `DsTypes::KERNEL`; its only caller is a standalone tool, not the compile pipeline. `OpFuncs::AllGather` is real but multi-AIU/cross-chip (`comm_size`, `Rank_t`, `world_size`) — out-of-lane, not a single-AIU LX primitive. None of these gaps is "the coordinate contract can't express it." Every one is "the backend op that would consume the coordinates isn't wired to this ds-type / this caller." The coordinate contract was **sufficient and un-exercised**, not insufficient. That is a strong refutation of the "Torch must own the physical plan" premise for this lane.

**Must be expanded on two axes — confirmed.** Coordinate metadata says *which core owns which slice*. It says nothing about (a) stick-**form** and (b) a reduction-**op**. Both are genuine gaps:

- **Form field (restickify) — confirmed needed.** The dldsc-lane demonstrates the gap by sidestepping it: `LXRelayoutPlan` has no pre/post layout field, so the prototype classifies `layout_restickify_activation` with `realized=False` and instead does a pure **opfunc string swap** (`ReStickifyOpHBM` → `ReStickifyOpLx`) gated on all-args-LX-resident. That works only because "the pre/post layout is already implicit in the restickify SDSC's own operand descriptors" — i.e. it relies on the *op* carrying the form, precisely because the *plan* cannot. The deeptools check confirms `ReStickifyOpLx` is op-present but contract-absent: `insertRelayoutSdsc` never *chooses* it and assumes same-layout input/output LDS. So to *plan* (rather than rename-and-hope) a layout-changing move, the interface needs source-layout, dest-layout, and operand identity. That is a real interface expansion.

- **Reduction-op field (reduce/all-reduce) — confirmed needed, and correctly out-of-scope to *design*.** The coordinate contract has no arithmetic. On-chip reduce exists only as PSUM/array-local DL compute (`SUM/MAX/MEAN`); the cross-device `AllReduce` is a different subsystem. Neither is expressible as "move these coordinates," because the many:1-with-arithmetic case needs an op tag the coordinate delta can't imply. State it as a future field; don't design it now.

## 3. Too much / too little to the backend? SLIGHTLY TOO MUCH IN THE FRONTEND TODAY — and it's the removable part.

The interface leaves realization to the backend correctly. The mis-balance is the *other* direction: the frontend currently does work the backend should own — hand-classifying `communication_class`/`communication_pattern` per collective. That is the part that will not scale (see §4). Strip it, and the frontend/backend balance is right: frontend detects + supplies coordinates (+ future form/op fields); backend derives cardinality and realizes.

## 4. Does it scale, or does each class force a bespoke path? PURE-MOVEMENT SCALES ON ONE PATH; the current per-class labeling does NOT — and that's the thing to cut.

The dispositive evidence: **one** backend path (`insertRelayoutSdsc → STCDPOpLx`) already covers scatter *and* multicast/broadcast from the same coordinate maps. That is the scalability proof — the pure-movement axis does not fork per class. The failure mode to avoid is exactly what the stcdp-lane started toward: a Python-side taxonomy where each cardinality gets its own class, its own `communication_pattern` string, its own realization assumption. That forces a bespoke path per class *in the frontend* and drifts from the single backend path that actually generalizes. Form-change and reduce genuinely need new *fields* (and, for reduce, likely a distinct backend op) — but those are two field additions, not N per-class paths.

## 5. Where I'd draw the contract differently, and what it buys

**One move:** make the coordinate delta the *sole* input for pure movement, and demote `communication_class` to a backend-emitted annotation (or drop it). The frontend supplies two distributions + a "move-worth-it" flag; the backend classifies cardinality and lowers. **This buys** (a) elimination of frontend/backend classification skew — no more "0 rows fired because the class label didn't match the edge shape," (b) automatic coverage of any new cardinality the coordinate maps can express (all-gather included) with zero frontend change, and (c) a single audit point for correctness.

**Two additive fields, not new lanes:** a `stick_form{src,dst}` + operand-identity field for restickify, and a `reduce_op` field for the arithmetic collectives. The backend consumes form via the already-live `ReStickifyOpLx`; the reduce field waits on a real move-then-reduce primitive (absent today — flag, don't design).

**Net:** the production split is architecturally sound and the DLDSC coordinate interface is the right spine. The correction is to push cardinality classification *down* to the backend (it was never needed up top) and to grow the interface by exactly two orthogonal fields to span the taxonomy — rather than growing it by one bespoke class per corner of the lattice.

**Status honesty:** scatter is implemented + measured (~1.19x, device-verified). Broadcast/all-gather is classified + prototype only (0 classifications fired on full Granite; the one realized operand probe fails in DXP on a `KERNEL` vs `INPUT` backend gap). Restickify-lx is classified-only + a guarded opfunc-swap prototype, unmeasured. Reduce/all-reduce is classified conceptually, absent as an in-lane primitive. None of the three adjacent corners is done.
