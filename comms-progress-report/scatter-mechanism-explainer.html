Keeping activations on chip: the scatter relayout

How the production path removes a class of HBM round-trips by moving a tensor core-to-core instead of spilling — the mechanism, the alternatives we didn't take, and where it can go next.

The accelerator has a small per-core scratchpad (LX) and slow main memory (HBM). When a producer op leaves a tensor resident in LX under one core work-division, and the consumer wants that same logical tensor under a different core ownership, the simple compiler writes it out to HBM and reads it back. That round-trip is pure overhead — the values don't change, only which core owns which slice. The scatter relayout removes it: the tensor stays in LX and the pieces that need to change owner move directly over the on-chip ring.

How it works: the frontend records, the backend synthesizes

The design splits cleanly. The frontend (Torch/Inductor) knows why each op picked its work-division and whether an edge is worth keeping on chip — that is policy. The backend (deeptools) knows the LX memory state, piece sizing, and legality — that is realization. So the frontend records what is needed and the backend decides how.

Torch owns the movement intent. The backend owns the movement implementation — because the implementation depends on LX memory state that only the backend can see.

Honest status. The scatter path is implemented and the backend synthesis is live. But the measured block win (~14.70 → ~12.34 ms/iter) comes from keeping fused-chain intermediates LX-resident, not from named relayout lowering firing — on the full block the collective classifications did not fire and the explicit HBM restickify rows are unchanged. Scatter's edge coverage is high; its isolated latency contribution is not yet measured.

Alternatives, and why not

Three other ways to bridge a work-division mismatch. Each was considered and set aside for the scatter corner for a concrete reason.

alternativewhat it iswhy not (for scatter)
Explicit frontend movement rowsTorch enumerates every source→dest core pair and emits the transfer rows itself.Duplicates what the backend already derives from coordinates, and hard-codes piece sizes / core pairing / LX-vs-HBM fallback that depend on backend memory state Torch cannot see. Only justified where the backend can't derive the move — which is the all-gather lane, not this one.
Co-assignmentForce producer and consumer onto one shared work-division so there is no mismatch.Couples two ops' division choices; the producer's best split and the consumer's best split generally differ, so this de-optimizes one op's array fill to save a movement whose cost is far smaller than the compute penalty. Also can't express fan-out.
Just spillThe status quo: write to HBM, read back under the new view.Always correct, but it is the round-trip being removed. Kept only as the backend's fallback when the resident piece doesn't fit in LX.

Where it can go next

  • Backend-derived cardinality. The move is already derived from the two ownership maps; the class (1:1 scatter vs 1:many broadcast vs many:1 gather) is also implied by those maps. Deriving it in the backend would let the frontend drop per-collective labeling and collapse to pure detection — one path covers every cardinality the coordinates can express, instead of one bespoke label per class.
  • Ring-distance-aware placement. Piece assignment ignores ring topology today; placing pieces to minimize hop distance between producer and consumer cores cuts on-ring latency for the same logical move.
  • Overlap with compute. The relayout is a separate scheduled step. Streaming it while the producer's tail or an unrelated op runs would hide the move under compute instead of serializing it.

The first of these is the important one: it is not just an optimization but a contract simplification. If the backend derives cardinality, then broadcast, gather, and all-gather need no new frontend path — only the two genuinely different axes (a layout-form change, and arithmetic reduction) call for the interface to grow, and each by a single field rather than a new lane.
