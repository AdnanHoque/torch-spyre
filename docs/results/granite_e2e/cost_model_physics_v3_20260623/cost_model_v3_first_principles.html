1 · Hardware First Principles: the weight-stationary array and the split

Everything the cost model decides — every penalty term, every tie-break — is a consequence of one piece of silicon: the per-core weight-stationary PT array. Understand how that array eats a matmul and the rest of the model stops looking like a pile of magic constants and starts looking like a description of physics.

The unit of compute: one core's 8×8 weight-stationary array

An AIU has 32 cores. Each owns one 8×8 weight-stationary PT array (the MPE — matrix-processing engine). For a matmul [B,M,K] @ [B,K,N] -> [B,M,N], the array does not stream both operands. It loads a weight tile [K,N] and holds it resident; then the M dimension — the tokens, the rows — streams past that resident weight, one batch of _PT_ROWS = 8 rows at a time.

This asymmetry is the source of every instinct in the model:

  • Loading the weight is expensive (fill the array's registers, drain the pipeline at the end).
  • Streaming a row is cheap (it just flows through the systolic fabric).

So the array is efficient exactly when the cheap thing (streaming rows) dominates the expensive thing (load + drain). You make that true by streaming a lot of rows past one weight load.

"Array fill": rows-per-core vs PT passes

When the planner splits M across m cores, each core sees only a slice. Define the per-core M tile:

m_t       = M // m                         # rows this core's array streams
pt_passes = max(1, m_t / _PT_ROWS)          # how many 8-row passes that is

pt_passes is the real currency: how many times the cheap streaming step happens per expensive weight load. The model's target is _TARGET_PT_PASSES = 5 — about 5×8 = 40 rows/core keeps the pipeline comfortably full. The compute term turns this directly into a derate:

pt_eff     = min(1.0, (pt_passes / _TARGET_PT_PASSES) ** 0.25)   # exponent 0.25
compute_us = (B*M*N*K / cores_used) / (_PEAK_MACS_US_CORE * pt_eff)

The 0.25 exponent says the loss is sub-linear: startup/drain is fixed, so even a moderately short tile recovers most of peak, but a 1–2-pass tile is genuinely starved. (The baseline uses 0.5 / sqrt against _TARGET_PT_PASSES = 8 — same tile, different grade. We are pricing v3.)

Worked on a real Granite decode attention@V shape, B=32 heads, M=64, N=128, K=576, true BMM:

splitm_t = M/mpt_passespt_effwhat the array sees

{b:8,m:4,n:1,k:1} (v3)64/4 = 162.00.8016 rows/core — array fed
The single most important intuition Same 32 cores in both rows (8×4 = 32 = 1×32). The win is not more cores — it is a healthier per-core tile: pulling rows-per-core from 2 up to 16 by spending the independent head-batch axis b instead of over-shredding M. This falls straight out of "weight loads are expensive, row streams are cheap."

The stick: the unit of N and K (never M)

Memory on Spyre is quantized into sticks: a 128-byte aligned, 128-byte contiguous block — at fp16 (_DTYPE_BYTES = 2) exactly 64 elements. Which dimensions are sticks follows the matmul layout the runtime uses: for C[m,n] = A[m,k] @ B[k,n], A is sticked on k, while B and C are sticked on n. So:

  • K is a stick dimension — contiguous/reduction axis. K=576 → 9 sticks; 512 → 8; 128 → 2; 64 → exactly 1.
  • N is a stick dimension — output/weight column axis. N=4096 → 64 sticks.
  • M is the streaming dimension and is never a stick. M is counted in rows and PT passes.

This is why decode QK^T's v3 pick {b:4,m:4,n:1,k:2} keeps the per-core K tile at 64 elements — a k=2 split of K=128 leaves 64 = one clean stick per core, exposing reduction parallelism without fracturing a stick.

The iteration-space split: B × M × N × K over 32 cores

The planner chops the iteration space into B_split × M_split × N_split × K_split. The notation 1_4_8_1 means B1 M4 N8 K1 — a product of 1×4×8×1 = 32 cores. The non-obvious truth: all four axes can multiply to 32, but they stress completely different hardware limits. "Use all 32 cores" is necessary but nowhere near sufficient.

HBM broadcast cohorts: why N-splitting isn't free

The set of cores sharing one broadcast is a cohort. The broadcast link has a soft capacity of _COHORT_LIMIT = 8: up to 8 cores share cheaply; past that they contend.

fanout_split   = max(m, n) if shared_weight else n
cohort_penalty = max(1.0, (fanout_split / _COHORT_LIMIT) ** 0.75)
hbm_us         = bytes_total / (_HBM_BW_GBS * 1000) * cohort_penalty
  • What fans out depends on whether the RHS is shared. For a shared-weight projection both M- and N-cores fan the operand out, so fanout_split = max(m,n). For a true BMM the weight is different per batch item, so only the N-split fans out: fanout_split = n. This is exactly why head/batch parallelism is cheap for attention — spending b doesn't raise the cohort at all.
  • Shared weights are counted once. weight_batches = 1 if shared_weight else B, so a projection weight is charged once, a true-BMM weight B times.

The PSUM ring: why K-splitting costs (k-1) hops

Split K across k cores and each computes a partial sum; (k-1) ring hops fold them into one.

psum_coeff            = _PSUM_PER_CORE_ELEM_US if shared_weight else _BMM_PSUM_PER_CORE_ELEM_US
output_elems_per_core = (B*M*N) / max(1, b*m*n)
psum_us               = max(0, k-1) * output_elems_per_core * psum_coeff

The crucial detail is output_elems_per_core: each core reduces only its own tile, not the whole output. Charging the per-core tile is what lets a small, well-shaped K-split (decode QK^T's k=2, one 64-element stick/core) win when it improves the tile.

The substrate the rest of the model stands on Strip away the constants and the picture is physical: (1) each core is fast only when many M rows stream past one resident weight load — that is pt_passes, the heartbeat; (2) memory is 64-element fp16 sticks — N and K stick, M streams; (3) the split is a four-way tradeoff, not a core count. Every penalty in _matmul_split_cost is one of three pressures — array fill, broadcast fanout, PSUM hops — written down as microseconds.

2 · The three physical-cost terms: compute, HBM, PSUM

Every split is first scored by three terms that model what the hardware physically has to do: multiply-accumulate, move bytes, and combine partial sums. These are the floor of the cost — the irreducible work. Everything else is a shape correction layered on top.

The 9-term additive v3 score

return (
    compute_us              # 1 · MACs over an M-derated peak
  + hbm_us                  # 2 · bytes / bandwidth, weight once, cohort-derated
  + psum_us                 # 3 · (k-1) ring hops, charged PER CORE
  + m_lane_underuse_us      # 4 · too few M lanes (tie-break)
  + m_tile_underfill_us     # 5 · per-core tile too shallow  <-- the decode fix
  + wide_n_us               # 6 · per-core N strip too wide
  + large_m_tile_shape_us   # 7 · gated shape terms  <-- the prefill fix
  + core_underuse_us        # 8 · soft replacement for the hard guard
  + batch_split_us          # 9 · additive, true-BMM only
)

Terms 1–3 are physics; 4–9 are guard rails (§3) and the regime gate (§4). We build the three physics terms first.

term 1 · physics

compute_us

pt_passes = max(1, m_t/8) pt_eff = min(1,(pt_passes/5)**0.25) compute_us = (B·M·N·K/cores)/(PEAK·pt_eff)

Per-core MACs over an M-derated peak. _PEAK_MACS_US_CORE = 1,536,000 MACs/µs/core. Peak is conditional: pt_eff is the fraction kept when the M tile is short. Worked: decode attn@V — main 2 rows/core → pt_eff 0.669 → 4.59 µs; v3 16 rows/core → 0.795 → 3.86 µs (−16%).

term 2 · physics

hbm_us

weight_batches = 1 if shared else B bytes = (B·M·K + wb·K·N + B·M·N)·2 fanout = max(m,n) shared else n cohort = max(1,(fanout/8)**0.75) hbm_us = bytes/204800 · cohort

Bytes over bandwidth. Weight counted once for shared_weight — the single most important HBM correction. Cohort penalty is sub-linear. Worked: MLP-down 122.16 MB → 596.48 µs; both {m:4,n:8} and {m:8,n:4} have fanout 8 → cohort 1.0, so they tie.

term 3 · physics

psum_us

coeff = 1e-3 shared / 1e-4 bmm oepc = (B·M·N)/(b·m·n) psum_us = max(0,k-1)·oepc·coeff

Zero unless K-split. Per-core, not whole-output — the decisive correction. Worked: decode QK^T k=2: baseline whole-output = 165.15 µs (forbids the split); v3 per-core = 7.37 µs (permits it). A 22× gap.

1 · compute_us — why peak is conditional, and why v3 lowered the exponent 0.5 → 0.25

Pipeline fill is a diminishing penalty — going 1→2 passes helps a lot; 4→5 helps a little. A power law with exponent < 1 captures that concave shape. The baseline used **0.5 against _TARGET_PT_PASSES = 8; v3 uses **0.25 against 5. v3's derate is gentler and saturates sooner (at 40 rows/core vs the baseline's 64), so it stops punishing tiles for being slightly short.

Worked — MLP-down prefill, shared 1×512×4096×12800. Total MACs = 2.684e10; per-core = 8.39e8. Both {m:4} (m_t=128) and {m:8} (m_t=64) clear saturation, so pt_eff = 1.0 and compute_us = 546.13 µs identically. Compute alone cannot separate these two — which is exactly why the shape terms exist.

2 · hbm_us — weight counted once, cohort sub-linear

Worked — decode attn@V true-bmm. bytes_total = 7.60 MB → hbm_us = 37.12 µs. Both main {b:1,m:32} and v3 {b:8,m:4} have fanout = n = 1 → cohort_penalty = 1.0: for a true BMM, head/batch and M parallelism cost zero broadcast contention. The baseline's max(m,n)/8 would have charged main's m=32 a 4× penalty, falsely making the starved pure-M split look bandwidth-bound when its real problem is compute underfill.

3 · psum_us — partial sums charged per core

Two coefficients: shared-weight 1.0e-3 vs true-BMM 1.0e-4 (10× cheaper). Attention BMMs frequently want a small K-split to get the tile geometry right (decode QK^T's k=2 → one 64-element stick/core), and the lower coefficient reflects that the fused-attention ring reduction is a measured, well-tolerated path there.

The throughline The baseline's three physics terms each had a structural over-charge — whole-output PSUM, unconditional batched weight, linear max(m,n) cohort — that systematically pushed it toward the same lazy pure-M 32-core picks. v3 doesn't add cleverness; it makes each one physically honest. Honest physics is enough to stop the bad picks. The shape terms exist only to break the ties that honest physics correctly reports as ties.

3 · The tie-break and guard-rail terms

Physics alone systematically picks the wrong split, because the generated kernel pays costs the analytic model can't see: PT startup/drain on short tiles, schedule overhead on wide output tiles, layout/fusion fallout from carving a dimension that didn't need it. Five guard-rail terms encode each as a penalty that is one-sided (punishes the unhealthy direction, exactly zero in the healthy one) and, for batch, additive.

The design philosophy A guard rail should make a genuinely bad tile lose, and be invisible to a healthy tile — never tax a good split just for being large.

The failure mode being fixed: the baseline's multiply and guard

The baseline returns (compute_us + hbm_us + psum_us + target_m_us) * batch_penalty. Two things cause its bad decode picks:

  1. batch_penalty = b ** 1.4 multiplies the entire score. For a true BMM where each batch item has tiny M, batch parallelism is exactly the right lever — but the multiplier makes it categorically unaffordable.
  2. The hard trade-down guard (if math.prod(new_splits) < math.prod(splits): return splits) discards any cost-argmin using fewer than 32 cores and returns the pure-M default. Even when the cost function correctly scores a 16-core split as best, the guard overrides it. The single biggest cause of bad decode/attention picks.
term 4 · tie-break

m_lane_underuse_us

target_m = clamp(M//(4·8), [4,16]) = max(0, log2(target_m/m)) · 10

Watches the number of M lanes (m), not the tile. Too few M lanes → streaming dim under-exposed. Worked: decode QK^T baseline m=1, target_m=4 → 20 µs; v3's m=4 → 0. Intentionally cheap (10/step) — a tie-break, not a hammer.

term 5 · THE decode fix

m_tile_underfill_us

= max(0, log2(16/m_t)) · 30

Wants each lane's tile deep enough. Strictly one-sided: 0 for every m_t≥16. Fixed-magnitude (30/step) so it survives in the low-compute decode regime where the pt_eff derate is too quiet. Worked: decode attn@V 2 rows/core → log2(16/2)·30 = 90 µs rejection; v3's 16 rows → 0.

term 6 · guard

wide_n_us

n_t = N//n = max(0, log2(n_t/512)) · 25

A per-core N strip too wide loses schedule efficiency. Mirror of underfill: that protected the streaming dim from being shallow, this protects output from being wide. One-sided above 512. Worked: prefill MLP-up n=8 → n_t=1600 → ~41 µs; n=4 → 3200 → ~66 µs (pushes toward more N lanes). Decode N tiles (128–576) score 0.

term 8 · soft guard

core_underuse_us

= max(0, log2(32/cores)) · 150

The soft replacement for the deleted hard guard. 150 is deliberately large — 16 cores costs 150 µs — so the planner still prefers full occupancy. The difference is at the margin: a 16-core tile that's >150 µs healthier can now win. A slope, not a cliff.

term 9 · additive, true-BMM only

batch_split_us

= 0 if shared else log2(b)·10

Additive and logarithmic instead of multiplicative. Worked: b=8 → baseline 8**1.4 = 18.4× (scales whole score); v3 log2(8)·10 = 30 µs (added once). The 30 µs is trivially outweighed by the 120/90 µs of underfill it buys back. Gated to true BMMs.

4 · The M-fill gate (the heart of v3)

Everything up to here has been about one failure mode: starving the PT array. The intermediate b473bff "physics" model fixed decode (kernel 14.765 → 10.621 ms/iter, 1.39×, reproduced in-block) but left prefill flat and on some attention BMMs slightly worse. The resolution is one boolean gate, filled_m_tile_factor, that turns a family of shape penalties on only when the array is already full.

Why b473bff fixed decode but not prefill

v3 is b473bff plus exactly one term, large_m_tile_shape_us. That makes the diagnosis sharp: every term b473bff already had is a floor — it punishes too little work per core. None is a ceiling. None notices when a split has passed the point of usefulness and started to hurt via downstream layout/fusion fallout.

  • Decode (M=64): the array is starved. Almost any move off pure-M raises rows/core toward target. The floor points the right way.
  • Prefill (M=512): the array is already full. The floor terms go quiet, and with no opposing force the argmin wanders into output-splits the array never needed — reshaping the per-core tile in ways that cost the fused kernel (layout, fanout, transpose epilogues) without buying compute.

The gate

filled_m_tile_factor = 1.0 if m_t >= _M_TILE_UNDERFILL_TARGET else 0.0   # target = 16

This is the same threshold the floor term m_tile_underfill_us uses — 16 rows/core = 2 PT passes. That is not a coincidence; it is the whole trick. 16 is the dividing line between "starved" and "fed":

  • Below 16: m_tile_underfill_us is positive (floor biting) and the gate is off (ceiling silent).
  • At/above 16: m_tile_underfill_us = 0 (floor satisfied) and the gate is armed.

What the ceiling penalizes: large_m_tile_shape_us

A sum of three geometry-driven sub-terms — driven by ratios of dimensions, never op names. Two are gated by filled_m_tile_factor; the third is not.

(1) true_bmm_value_split_us — true BMMs with K >> N

Recognizes a true BMM whose reduction dim K is much larger than its output N — the value-matmul shape. Penalty scales with both lopsidedness (log2(K/N)) and how hard you split the tiny N (log2(n)). Zero if n≤1.

A 40–120 µs wall keeps the argmin on {m:32,n:1}. In the decode counterpart ({b:8,m:4}, m_t=16) the gate is also armed, but n=1 so the term is 0 — the decode win came from the floor and batch_split, not the ceiling.

(2) shared_narrow_tile_us — shared projections with too-wide per-core N

Recognizes a shared narrow projection (N<4096) whose per-core tile (n_t = N/n) is still wider than 128. Half-strength coefficient (5.0) — a gentle nudge toward more N-splitting, the opposite direction from sub-term (1). Worked: prefill K/V proj N=1024, narrowness log2(4096/1024)=2; n=4 → n_t=256 → 2·1·5 = 10 µs. It's one voice among nine — the v3 K/V pick stays {m:8,n:4} because the other terms outvote a 10 µs nudge.

(3) shared_down_n_split_us — shared K>N down-projections (UNGATED)

The one piece not multiplied by filled_m_tile_factor. The artifact rationale: b473bff was "still slightly too eager to split N in some folded shared-RHS cases," and that eagerness showed up even where M wasn't large — so gating it on m_t≥16 would leave those unfixed. Worked: prefill MLP-down K=12800, N=4096, log2(K/N)=1.64; n=4 → 32.9 µs, n=8 → 49.3 µs. The ~16 µs step keeps the long-K down-proj off an N-split; the v3 SDSC pick moves to {m:8,n:4,k:1}.

The central intuition

The heart of v3 Not a new axis, not more cores, not an op-name table — one boolean, filled_m_tile_factor = 1.0 if m_t >= 16 else 0.0, placed at the precise hardware boundary between a starved PT array and a fed one. Below it, push work outward. Above it, stop. The decode win (1.39× in-block, 274→177 ms e2e) comes from the floor; the prefill restoration that makes 335 ms hold (rather than regressing as b473bff did) comes from the gated ceiling. Same model, two regimes, one threshold.

5 · Baseline vs v3: the canonical model and its three structural problems

An upstream "default distributor" has already proposed a 32-core pure-M split for every matmul. The cost-model planner is asked whether it can do better. Baseline and v3 use the same search loop — enumerate every legal B×M×N×K product ≤ 32, score, take the argmin — but two very different scoring functions, and two very different rules about whether the argmin is allowed to win.

Problem 1 — the multiplicative batch funnel

The batch penalty doesn't add a cost; it scales the entire estimate. On the canonical decode attn@V shape:

splitrows/corePT statebaseline score
{b:1,m:32,n:1,k:1} (pure-M)2starved307.2 µs
{b:8,m:4,n:1,k:1} (head-batch)16fed2633.1 µs
Problem 1The head-batch split gives each core 8× more useful work and yet scores 8.6× worse — because b=8 multiplies the whole sum by 8**1.4 ≈ 18.4. The one knob that fixes decode array-fill is exactly the knob the funnel forbids. v3 fix: additive structure; b=8 costs a flat +30 µs, and v3 scores the head-batch split at 71.0 µs vs pure-M at 131.7 µs — the healthy tile wins by the ~1.9× the device actually shows.

Problem 2 — whole-output PSUM over-penalizes useful K-splits

The baseline charges the whole output per ring hop: psum_us = (k-1)·(B·M·N)·1.4e-4. For decode QK^T even a single k=2 hop bills 165.2 µs — independent of how the K-work was spread. v3 fix: per-core (k-1)·(B·M·N)/(b·m·n)·coeff, with coeff 1e-3 shared / 1e-4 true-BMM. The decode QK^T pick {b:4,m:4,n:1,k:2} can finally afford its clean one-stick reduction split.

Problem 3 — the hard trade-down guard (the big one)

After the search, the baseline planner does if math.prod(new_splits) < math.prod(splits): return splits: if the cost-model's split uses fewer than 32 cores, discard it and return the pure-M default — a hard veto with no appeal to the cost it just computed.

The false axiomCores are not the scarce resource — array fill is. A split that uses all 32 cores but hands each one 2 rows leaves every array mostly draining. The guard's "more cores = better" is precisely backwards for small-M decode, where using fewer cores with more rows each is the win.

The throughlineThe baseline encodes its array-fill intuition in two places that fight the hardware — a multiplier that punishes the batch axis decode needs, and a guard that worships the core count decode should sacrifice — while measuring K-reduction and shared weights too crudely to recover. v3 moves every one of those intuitions inside a single additive score as a soft, regime-gated term, so the planner can finally pick the split that fills the PT array rather than the split that merely lights up all 32 cores.

6 · Evidence: picks, oracle gaps, and the e2e result

Four independent witnesses, in increasing authority: (1) the 12-shape pick table (the model's mind changed); (2) the isolated-matmul oracle (how wrong was main, on device); (3) the one-layer fused block kineto (do the changes survive fusion); (4) Antoni's full Granite e2e (the aggregate source of truth, and the only witness that carries the prefill win).

DisciplineNever let a weaker witness speak for a stronger one. The block does not get to claim the prefill e2e number. The pick table does not get to claim a speedup. Each says exactly what it measured.

The prefill win is real per Antoni's e2e (495 → 335, ~1.48×, approximate denominator), and only the e2e carries it. The block must never be cited as its source.

Bottom line v3 is not a cleverer model — it is an honest one. Every baseline structural over-charge (whole-output PSUM, unconditional batched weight, linear cohort, multiplicative batch, the hard core-count guard) systematically funneled the planner toward the same lazy pure-M 32-core picks that starve the weight-stationary array in the small-M decode regime. v3 makes each pressure physically honest and additive, then adds one boolean — filled_m_tile_factor at m_t = 16 — that lets a single cost function give prefill and decode opposite instincts: feed the array, then stop. Result: best reported Granite e2e to date (335 ms prefill / 177 ms decode, ~1.5× over the baseline reference), with decode strongly multi-witnessed and prefill carried by the full e2e.
