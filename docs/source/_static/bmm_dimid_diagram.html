Why the cost-model planner declines a shared-weight (3D @ 2D) bmm

The production MLP lowers to a batched matmul with a shared 2D weight. The work-division cost model (Pass 2) currently can't handle that shape and hands it to the default distributor (Pass 3). Here is exactly why.

The one idea: the planner figures out which dim is M (the matmul's row dim) by looking for the dim that appears in only one input. A normal batch dim sits in both inputs, so it's easy to tell apart from M. But a shared 2D weight is broadcast across the batch with stride 0 — which makes the batch dim disappear from the weight's coordinates. Now two dims (batch and M) both "appear in only one input," the planner can't tell which is M, and it bails.

Why the 3D @ 2D case is the hard one — step by step

The trouble is that a shared weight is genuinely a broadcast (the same weight reused for every batch element), and the lowering represents that broadcast in a way that erases the exact signal the dim-identifier relies on. Walking it through:

  1. The MLP produces a 2D @ 2D matmul, which gets rewritten into a bmm so it can be batched: _unflatten_mm_to_bmm turns mm(x_2D, W_2D) into bmm(x_3D, expand(unsqueeze(W))) (temp_passes.py:41-45,101). The result is a 3D @ 2D-in-spirit op: a real 3D activation times a weight that is the same across the batch.
  2. The broadcast is a stride-0 view, not a copy. expand gives the weight a batch dimension with stride 0 — there is no per-batch weight data, just the one weight read repeatedly. This is correct and is exactly what we want (read the weight once).
  3. Stride-0 dims are dropped from coordinates. The coordinate system intentionally omits stride-0 dims because they carry no addressing (compute_coordinates, views.py:91,108). So the batch dim has coefficient 0 in the weight's index and never appears in the weight's device_coords.
  4. Dim-ID separates batch from M using "appears in both inputs." The identifier assumes a batch dim shows up in both operands and M shows up in only the LHS, then takes M as the unique single-input row dim (m_candidates = [d for d in m_dims if _n_inputs(d) == 1], work_division.py:856).
  5. For the shared weight, that assumption is false. Because step 3 removed the batch dim from the weight's coords, the batch dim now appears in only the LHS — exactly like M. So m_candidates holds two dims (batch and M), len != 1, and the planner returns None (work_division.py:857-858), caught as if dims is None: return splits (work_division.py:1218-1219) — the op falls through to Pass 3.

The deeper reason it's "complex": from the coordinates alone, a broadcast-batch bmm is indistinguishable from a plain 2D matmul that happens to have an extra leading dim. The information that disambiguates them — "this operand is a rank-2 weight that was broadcast" — has been thrown away by the time dim-ID runs. Recovering it needs weight-rank awareness (knowing the weight was broadcast), not just coordinate inspection.

The fix (follow-up)

Make dim-ID recognize the broadcast: detect the stride-0 weight (or carry the original weight rank through the lowering) so the batch dim is distinguishable from M again. Once shared-weight bmms reach the cost model, the B·K·N weight term should also be corrected to K·N for a shared weight so the broadcast's HBM cost isn't over-counted.
