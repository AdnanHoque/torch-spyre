Closing the torch-spyre ↔ sendnn gap

What the remaining matmul/MLP gap actually is, how to reach parity, and how to beat sendnn — grounded in measured kernel decompositions, not estimates. Companion to the MLP deep-dive.

The one line: on the latest main (5838b3b) the tsp↔sendnn gap is a measured kernel-codegen problem, dominated by PT-array under-fill — tsp's wide-N MLP matmul fills the 8×8 systolic array to only 29.5% vs sendnn's 77.7% (a 3.3× kernel gap) — plus SiLU not fused into the matmul epilogue and the FFN not fused to one kernel. None of it is in the work-division planner: PR #2407 already picks good splits (it cut prefill 832→495 ms). The remaining prize once the array is filled: warp-specialized inter-matmul pipelining, the on-chip ring work already prototyped on attention.

The single most important diagnostic: PT-Util 29.5 vs 77.7
PT-Util is how fully each kernel fills the 8×8 systolic (PT) array. On the wide-N prefill matmul, tsp fills it to only ~38% of sendnn's occupancy (29.5/77.7). That reciprocally matches the 3.30× kernel gap — under-fill the array and compute takes proportionally longer. This is a codegen/tiling problem, not a planner problem: the work-division planner decides how many cores and which axis to split; it does not decide how the wide N=12800 dimension tiles onto the array inside one core's kernel. PR #2407 correctly cut whole-model prefill 832→495 ms by choosing good splits, but it cannot touch the 3.3× per-matmul gap — that lives in deeptools/Inductor kernel codegen.

Decode has no MLP gap. The bare matmul (M=1, one weight read, batch-amortized) is at parity-or-faster (0.83×). The standalone full-FFN *1.94× above is a measurement artifact — the standalone probe reloads all 402.7 MB of weights every call (~24 ms of Memset), inflating its "kernel" number. Re-measured on device with resident weights, the decode FFN is tsp 2.84 / sendnn 2.76 ms = 1.03× — parity, both at the 402.7 MB/~143 GB/s weight-read floor (flat across M=1–16). Cross-check: Antoni's own bare decode matmul is 0.72 ms, so a 16.6 ms compute FFN of three matmuls is impossible. So decode has no MLP lever — improving the matmul, fusing the FFN, or folding SiLU all do nothing at decode; the whole MLP story is prefill.

The levers, concretely (in priority order):

  1. Fill the PT array on the wide-N matmul (prefill, the dominant term) — tsp's [4096,12800] kernel drives the 8×8 systolic array to only 29.5% util vs sendnn's 77.7%. Better tiling of the wide N=12800 dimension onto the array closes most of the 3.3×. This is the biggest single win and it is kernel codegen (deeptools/Inductor), not work division — the planner already picks the right core split.
  2. SiLU in the matmul epilogue — run the activation on the SFP unit concurrently with the matmul, as sendnn does, instead of a serial pass after. Mostly ours (Inductor fusion); needs the matmul SDSC to expose an epilogue hook. Prefill only — at decode the intermediate is tiny and the FFN is already at parity, so this buys nothing there.
  3. Fuse the FFN to one kernel — removes tsp's intermediate HBM round-trip and enables (2). Inductor decides it; deeptools executes it. A prefill win; decode is already at the memory ceiling.
  4. Pipeline the chained matmuls (the beat lever, both stacks) — let the down-projection's weight prefetch overlap the front matmul's compute. sendnn doesn't do this, so it's a genuine lead, not parity. Deeptools / SDSC overlap — same family as the on-chip ring producer→consumer work already shown on attention. Only buys speedup after the array is filled (lever 1).
Strategic read
Closing to parity is not a work-division / cost-model problem — the planner already picks the right splits (PR #2407). It is kernel codegen, and the dominant term is PT-array under-fill on the wide-N matmul (a deeptools/Inductor tiling problem, now in scope). The SiLU epilogue is the clearest Inductor-side win and the primary decode lever.

Rolling up and applying the per-op tsp/sendnn ratios (calibrated so the sendnn estimate lands on the measured ~190 ms — the standalone-probe ratios over-state the in-model gap by ~37%), the 305 ms prefill gap splits roughly evenly: ~53% MLP, ~47% attention.

The gap is in both MLP and attention
The MLP-silu block is the single largest kernel (41% of prefill) and its matmul runs at 3.3× with 38% array fill. Attention SDPA carries the worst relative efficiency in the report (7.6×, PT-Util 3.1 vs 23.6). So the prefill gap is genuinely split across both op classes — neither the MLP-matmul nor attention can be treated as already-solved.
  • Prefill: MLP-matmul (lever 1: array fill) and attention SDPA are co-equal targets. Fixing the MLP-silu kernel's array utilization is the largest single line; the SDPA kernel is the worst relative loss.
  • Decode: the MLP is not in the decode gap at all. Re-measured on device with resident weights, the decode FFN is at parity (tsp 2.84 / sendnn 2.76 ms = 1.03×), at the weight-read floor; the standalone 1.94× is per-call reload inflation, not compute. So the decode gap (if any) is outside the MLP — attention / KV-cache / per-token overhead — and none of the MLP levers move it. (The whole-model decode 2.0× is unverified — no decode config in the authoritative report.)

Caveat: the whole-model breakdown is tsp-only; per-class sendnn is estimated via micro-probe ratios, not measured. The split is robust in direction (both MLP and attention are real losses) but ±10–15 points in magnitude. Whole-model PT-Util figures are on a different normalization than the micro-probes — only the tsp-vs-sendnn relative util (29.5 vs 77.7, 3.1 vs 23.6) is interpretable.
