1. Five-second orientation

When the Spyre compiler sees a matmul, it has to decide how to split the work across 32 cores. There are many possible splits and they don't all run at the same speed — the difference between a good split and a bad one can be 2-10× in kernel time.

The cost-model planner enumerates feasible splits, scores each one with a small formula that estimates kernel time in microseconds, and picks the lowest score.

Where it lives

torch_spyre/_inductor/work_division.py, in _matmul_split_cost (the cost formula) and _cost_model_matmul_planner (the search loop).

2. The hardware in one minute (no jargon)

Picture the Spyre accelerator as a workshop with 32 workstations. Each workstation has:

  • A small workbench (called LX scratchpad) — 2 MB of fast on-chip memory.
  • A specialized matrix unit (called the PT array) — an 8×8 grid that multiplies pieces of matrices.
  • A general-purpose vector unit for everything that isn't a matrix multiply.

Outside the workstations, there is:

  • A giant warehouse (off-chip DRAM, "HBM") with 128 GB of capacity.
  • A conveyor between the warehouse and the workstations — total bandwidth ~166 GB/s, shared by all workstations.
  • A ring network connecting workstations directly (used when cores need to exchange partial results).

Two facts you'll need to remember:

  1. The matrix unit eats 8 rows at a time. Feed it 8 rows, or 16, or 24 — it stays full. Feed it 4 rows and it's half-idle. Feed it 64 rows and it just chews through eight 8-row sweeps in a row.
  2. The workbench is tiny. 2 MB doesn't hold much. If a worker's job needs more data than fits, they have to keep walking back to the warehouse for pieces. Walking to the warehouse is slow. Working at the bench is fast. We want to minimize walking.

3. What is a "split"?

A matrix multiply takes two matrices and produces a third:

   A (M × K)  ×  B (K × N)  =  C (M × N)

Three sizes describe it:

  • M — number of rows of A and C
  • N — number of columns of B and C
  • K — the shared dimension (we sum K products to compute each output element)

To use 32 workers in parallel, we slice these three axes among them:

  • m = how many cores work on different M-rows
  • n = how many cores work on different N-columns
  • k = how many cores split the K-axis (they each compute a partial sum and combine later)

Constraint: m × n × k ≤ 32.

2× faster, on the same hardware, doing the same math. The cost model exists to pick the right one without trial and error.

5. The cost function — six terms

The total cost is the sum of six terms, all in microseconds:

total_us = (compute_us
            + hbm_us
            + psum_us
            + target_m_us
            + lx_pressure_us) × batch_penalty
           + redistribution_us

The planner enumerates every (m, n, k) split that's feasible (each factor divides the corresponding dimension; product ≤ 32 cores), scores each one, and returns the lowest.

Now let's walk through each term and explain exactly what it represents.

5.1 compute_us

How long the actual multiplication takes.

A matmul of size M × N × K needs M × N × K multiply-accumulates (MACs). When we split across cores, each core does:

per_core_MACs = M × N × K / (m × n × k)

Hardware peak: the published number is 300 TOPS (tera-operations per second) per card. We back out per-core per-microsecond MACs:

peak_MACs_per_core_per_us = (300 × 10¹²) / 2 / 32 / 10⁶ ≈ 4.7M
                              ↑           ↑    ↑
                       300 TOPS    each MAC   32 cores
                                   is 2 ops

So naïve compute time: compute_us = per_core_MACs / 4.7M.

The PT-pipeline twist

The matrix unit needs 8 rows of input at a time. If a core has only 4 rows of A, the unit runs half-empty: efficiency = 4/8 = 50%. Worse: the pipeline that streams rows into the matrix unit takes ~8 sweeps to fully warm up. So we model: 

PT_passes = max(1, per_core_M / 8)        # how many 8-row sweeps the core does
pt_efficiency = min(1.0, PT_passes / 8)   # need ≥ 8 sweeps to fully fill the pipeline
effective_peak = 4.7M × pt_efficiency
compute_us = per_core_MACs / effective_peak

So:

  • per-core M = 64 → 8 PT passes → 100% efficiency. Best case.
  • per-core M = 16 → 2 PT passes → 25% efficiency. Compute time is 4× worse.
  • per-core M = 128 → 16 PT passes → 100% efficiency (capped). Fine.

5.2 hbm_us

How long to move all the matmul's data through the conveyor from the warehouse.

Total bytes the matmul touches:

bytes = (M × K)  +  (K × N)  +  (M × N)    × 2 bytes/elem (fp16)
          ↑           ↑           ↑
          A           B           C (output)

Divided by the bandwidth:

hbm_us = bytes / 166 GB/s
The cohort twist

When 4 cores all want the same row of A, they don't load it 4 times — the warehouse broadcasts it once. Smaller cohorts pay the conveyor cost; bigger cohorts amortize it. But past a certain cohort size (~8), the broadcast hardware itself becomes a bottleneck. 

cohort = max(m, n)
cohort_penalty = max(1.0, cohort / 8)
hbm_us = bytes × cohort_penalty / 166 GB/s

This means splits like (m=32, n=1) (cohort=32) pay a 4× HBM penalty in the cost model, which correctly rules them out for big matmuls.

5.3 psum_us

Only fires when K-split is non-trivial. The cost of summing partial results across cores via the ring.

If we split K across k cores, each core computes a partial sum of the output. To get the final output, those partials have to be added together. They flow around the on-chip ring, hop by hop.

psum_us = (k - 1) × (M × N) × 4 × 10⁻⁴

Walkthrough:

  • (k - 1) hops needed to fully reduce k partial sums (chain reduction).
  • M × N output elements per hop.
  • 4 × 10⁻⁴ microseconds per element per hop, calibrated from device measurements.

For k=1 (the usual case), this term is 0.

5.4 target_m_us

A small tie-breaker. When other terms are equal, prefer the m-split that lands the per-core M near the PT sweet spot.

Two splits often have identical compute_us, hbm_us, psum_us. We need a way to break the tie that's consistent with what we know about the hardware. The PT unit prefers per-core M = 64 (= 8 PT sweeps × 8 rows per sweep), so we penalize log2-distance from that:

target_m = clamp(4, cores/2, M / (8 × 8))
m_dist = abs(log2(m / target_m))
target_m_us = m_dist × 50

For M = 512:

  • target_m = min(16, 512/64) = 8
  • m=8 → distance 0 → 0 µs penalty
  • m=4 → distance 1 → 50 µs penalty
  • m=16 → distance 1 → 50 µs penalty

This term is small on purpose — it only matters when nothing else does.

5.5 lx_pressure_us

When a worker's slice of B is bigger than the 2 MB scratchpad, they have to keep shuttling pieces from the warehouse. Charge for that.

Each worker holds their slice of B in scratchpad. Slice size:

per_core_RHS_bytes = K × (N / n) × 2

If that exceeds 2 MB, charge for the excess:

excess = max(0, per_core_RHS_bytes - 2 MB)
lx_pressure_us = excess × 5 × 10⁻⁶
Why this term was added

Without it, the model picked (m=8, n=4) for wide-N shapes like MLP — but the empirical winner is (m=4, n=8). Reason: bigger n-split halves the per-core RHS slice, which halves the shuttle traffic. The fix calibrates the coefficient so the term only flips the choice when the overflow is severe (~13× over LX), matching the device-measured break-even point.

shape (M, K, N)per-core RHS at (8, 4)LX excesspenaltycost-model picks
KV (512, 4096, 1024)2 MB00 µs(8, 4) ✓
QO (512, 4096, 4096)8 MB6 MB30 µs(8, 4) ✓
MLP (512, 4096, 12800)26 MB24 MB120 µs(4, 8) ✓

5.6 redistribution_us

When a matmul is part of a fusion bundle with a non-matmul op, any non-default split forces the output to be reshuffled across cores so the partner can consume it.

Consider an MLP fused as silu(matmul(x, gate)) × matmul(x, up). The two matmuls and the silu and the elementwise multiply all run in one fused kernel. If the matmuls use a non-default split, the matmul outputs sit on cores in the "wrong" layout for the partner op (silu/multiply), so the output has to be shuffled before the partner runs. Shuffling burns HBM bandwidth.

For matmuls inside a fusion bundle (only), we add:

output_bytes = B × M × N × 2
redistribution_us = output_bytes × 1 × 10⁻⁴      # fired when bundled AND split ≠ default

For an MLP-shape output (B=1, M=512, N=12800 → 12.5 MB), this is ~1300 µs — much larger than typical kernel savings. So the cost model only rewrites bundled matmuls when the kernel win clearly outweighs the bundle penalty (which empirically almost never happens in MLP fusions).

The batch_penalty multiplier

The whole sum gets multiplied by:

batch_penalty = 1 + 0.6 × max(0, b - 1)

Splitting batch over multiple cores is mildly penalized because each batch item is independent work that could be iterated sequentially instead. For b=1 (no batch split) the multiplier is 1.

6. Walked example: MLP-class matmul

Shape: M=512, K=4096, N=12800, B=1. Two finalists, both using all 32 cores: (m=8, n=4) and (m=4, n=8).

term(8, 4)(4, 8)why they differ
compute_us~57~57Same per-core MACs, both at full PT efficiency (per-core M = 64 or 128, both ≥ target 8 passes).
hbm_us~265~265Same total bytes, both have cohort=8 (no broadcast penalty).
psum_us00k=1 in both.
target_m_us050m=8 is exactly target. m=4 is 1 log2 step away.
lx_pressure_us12055(8,4) per-core RHS = 26 MB → 24 MB excess. (4,8) = 13 MB → 11 MB excess.
total442427(4, 8) wins by 15 µs

So the cost model picks (m=4, n=8). Measured kernel time:

  • (8, 4): 2.61 ms
  • (4, 8): 2.10 ms — 20% faster

The cost-model's predicted margin (15 µs) is much smaller than the measured margin (~500 µs). That's fine — we only need the model to rank splits correctly, not predict their absolute times. The ranking is what determines which split gets compiled.

8. What the cost model does not solve

The planner picks the best split among the splits the SDSC framework can express — m × n × k partitions of cores. There is a separate gap to sendnn on some shapes (~2× kernel time on MLP-class matmuls) that comes from the kernel template itself: how RHS tiles are streamed, how PT passes are double-buffered, etc. That's a kernel-template engineering problem, not a work-division problem, and is out of scope for this planner.

If you swap to a faster kernel template later, the cost model's split choices will benefit too — but the planner doesn't need to change to capture that win.
