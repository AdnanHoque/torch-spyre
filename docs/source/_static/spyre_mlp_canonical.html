The torch-spyre MLP: first principles, the measured gap, and the path to parity

A single canonical reference for the SwiGLU FFN on the IBM Spyre AIU — what the op is, how it lowers, what the numbers mean, where torch-spyre trails sendnn on the authoritative production build, and the concrete levers to reach and then beat it. All performance numbers are from Antoni's spyre-perf-suite run on torch-spyre main 5838b3b (Granite 3.3 8B), unless explicitly marked otherwise.

The one screen. Whole-model prefill is 2.6× sendnn (495 vs 190 ms kernel time); the matmul work-division planner (PR #2407) already did its job, cutting prefill 832→495 ms by picking good core splits. The remaining gap is kernel codegen, not work division. It splits roughly evenly between two op classes:
  • The MLP wide-N matmul (≈53% of the gap) runs 3.3× over sendnn from PT-array under-fill — it fills the 8×8 systolic array to only 29.5% vs sendnn's 77.7%, at a favorable memory cost. Not a bandwidth wall, not a planner problem, not an inter-matmul sequencing stall.
  • Attention SDPA (≈47% of the gap) is the worst single op at 7.6× (PT-Util 3.1 vs 23.6).
Decode has no MLP gap at all: re-measured on device with resident weights, the decode FFN is tsp 2.84 / sendnn 2.76 ms = 1.03× — parity, at the weight-read floor. (The standalone mlp probe's 1.94× is per-call weight-reload inflation, not compute; §3.2.) So all MLP effort is prefill. Path to parity: fill the array on the wide-N matmul + fold SiLU into the matmul epilogue + fuse the FFN to one kernel; beat = pipeline the chained FFN matmuls (sendnn doesn't, so it's a genuine lead).

1. What the MLP is

The benchmarked mlp op is one SwiGLU FFN block — the feed-forward half of a single decoder layer. Two parallel "front" projections lift each token into a 4×-wider space, a SiLU gate multiplies the up-projection, and one "down" projection contracts back:

gate = x @ Wg              # [M, 4096] @ [4096, 16384]
up   = x @ Wu              # [M, 4096] @ [4096, 16384]
h    = silu(gate) * up     # elementwise, [M, 16384]   (~free)
out  = h @ Wd              # [M, 16384] @ [16384, 4096]

It is one FFN block — not a layer (no attention, no norms), not a model (no layer stack). The standalone mlp probe uses intermediate = 4×emb = 16384 (weights total 402.7 MB); the production Granite 3.3 8B layer uses intermediate = 12800, which is the shape behind the wide-N matmul numbers below.

Two regimes are benchmarked, and the contrast between them is the whole story:

  • Prefill: M = 512 tokens at once. Total work ≈ 1.03×1011 MACs — compute-heavy, the array fills.
  • Decode: one token per step (the profiled case is a 4-sequence batch [4, 1, 4096], M=1 each). Total work ≈ 8.05×108 MACs, about 128× less than prefill — yet it takes roughly the same wall time, because it must still stream the same 402.7 MB of weights.

2. How to read the numbers

The two stacks fuse the FFN differently, which matters before you read any table. torch-spyre emits two compute kernels (front: sdsc_fused_bmm_mul_silu_0 = gate+up+SiLU+mul → the [M,16384] intermediate; then sdsc_fused_bmm_1 = down) plus a Memset. sendnn fuses the whole FFN into one bmm plus host↔device Memcpy. To compare "kernel time," sum torch-spyre's two rows against sendnn's one.

metricwhat it is
kernel_msThe comparison metric. Sum of the compute SDSC kernel durations — excludes Memset, Memcpy, and host launch/dispatch. This is the actual PT-array work, not wall time.
spyre_msTotal device time for the call: kernels plus per-call stickify, HBM staging, Memset, and host↔device Memcpy. "Everything the device did this call" — inflated by per-call weight uploads (§5.5).
PT-UtilThe measured ratio compute_floor / kernel_time — the fraction of the 8×8 array's MAC capacity actually used. At decode it is ~0 because there is almost no arithmetic; at prefill a low value reflects a genuinely under-filled array (the central diagnostic below).
pt_effA cost-model de-rating factor, ~1.0 at the split the planner chooses. It says the chosen split is full in the cost model's bookkeeping — distinct from the measured PT-Util, which is the realized array occupancy the kernel codegen controls.
wall / cpuHost clocks (Python perf_counter, self-CPU). Dominated by dispatch and .to(device)/.cpu() — not the "is the accelerator slow" signal. Ignore for kernel comparisons.
Use ratios, not absolute ms, for the standalone op probes. They carry ~4× fixed per-op overhead — the standalone mlp reads 21 ms vs the in-model MLP-SiLU kernel of ~5 ms/layer (202.4 ms across 40 layers). Only the tsp-vs-sendnn ratio is meaningful. Likewise, whole-model PT-Util figures are on a different normalization than the micro-probes — only the relative tsp-vs-sendnn util (29.5 vs 77.7; 3.1 vs 23.6) is interpretable.

4. Where the gap lives

Rolling the whole-model prefill up by op class: MLP ≈ 56% of tsp prefill time (silu 202.4 + down-proj 73.7 = 276.2 ms) and attention ≈ 44% (219.5 ms). But the gap attribution is not the time split — the standalone micro-probe ratios over-state the in-model gap by ~37%. Calibrating them down so the bottom-up sendnn estimate lands on the measured ~190 ms whole-model sendnn prefill, the ~305 ms prefill gap splits:

classshare of the 305 ms gap≈ ms
MLP (matmul + epilogue)~53%~162
Attention (SDPA-dominated)~47%~143
The gap is in both MLP and attention — not one bottleneck
Fixing only the MLP matmul leaves roughly half the prefill gap on the table, and vice versa. The MLP-SiLU block is the single largest kernel (41%) and its matmul runs at 3.3× with 38% array fill; attention SDPA carries the worst relative efficiency in the report (7.6×, PT-Util 3.1 vs 23.6). The unifying symptom across both is array under-utilization, not a single missing kernel. (The per-class sendnn split is estimated via micro-probe ratios — robust in direction, ±10–15 points in magnitude.)

Decode is a different story — there is no decode-MLP gap to chase. Re-measured on device with resident weights, the decode FFN is at parity (tsp 2.84 / sendnn 2.76 ms = 1.03×, §3.2), both at the weight-read floor; the bare GEMM is at parity too (0.83×). The standalone probe's 1.94× is per-call reload inflation, not compute. So decode has no MLP lever at all — neither work division (PR #2407 doesn't move it) nor kernel fusion buys anything; the FFN is already at the memory ceiling. All MLP optimization effort belongs in prefill.

5. Why — first principles

5.1 Decode is memory-bound weight streaming

At one token, a single row enters the 8×8 array — ~1 of 64 cells active, PT-Util ~0.15%. That is not idle bookkeeping; there is almost no arithmetic to do. The ~2.8 ms the kernel spends is streaming all 402.7 MB of weights: 402.7 MB / 2.83 ms ≈ 142 GB/s (tsp), ≈144 GB/s (sendnn) — both right on the device's real read ceiling, with essentially zero fixed overhead. Both stacks land within ~1% of each other and the ceiling: the signature of a shared hardware limit, not a software gap.

5.2 The real memory bus runs at ~half its spec

The 204.8 GB/s LPDDR5 figure is a sticker, not a delivered rate. Measured two independent ways with no matmul: a pure read-only reduction (x.sum()) plateaus at ~143 GB/s; a pure copy (y = x×2, one read + one write) plateaus at ~113 GB/s. The read stream is ~27% faster — writes contend for the bus. This is the real ceiling every weight-stream hits, and it does not collapse at decode (the decode kernel sits right on the 143 GB/s read rate).

5.3 Prefill is per-matmul array under-fill

The standalone wide-N matmul [1,512,4096]@[4096,12800] (N=12800 = 200 sticks) runs 3.3× over sendnn (tsp 3.16 vs sendnn 0.96 ms). It is not a bandwidth wall, and it is not an inter-matmul effect:

  • Not bandwidth. Both stacks read the same 100 MB weight. sendnn finishes near the ~0.70 ms read floor (100 MB @ ~143 GB/s); tsp takes ~4.5× that floor, at a favorable memory ratio (tsp moves ~0.40× the HBM traffic). If it were bandwidth-bound, tsp would land near 0.96 ms. It does not.
  • Not work division. Forcing every legal core split on device leaves the kernel within ~2–3% — no split moves it (the planner already picks the right one).
  • Not inter-matmul sequencing. The full-FFN gap (3.6×) is barely larger than the single-matmul gap (3.3×), so chaining the FFN's matmuls adds almost no penalty. The loss is per-matmul, repeated across the FFN.
The dominant prefill loss: the array runs at 29.5% vs sendnn's 77.7%
On the same matmul reading the same 100 MB weight, sendnn measures PT-Util 77.7% and torch-spyre 29.5% — sendnn keeps ~2.6× more of the 8×8 array busy. If the low number were merely "the read floor showing through," both would land in the same band; they do not. So the low tsp PT-Util reflects a genuinely under-filled array, a kernel-codegen / array-tiling problem (deeptools / Inductor), not work division and not the bus. sendnn's M-sweep confirms the other side: sweeping M 1→512 grows its single-matmul kernel only +0.24 ms — compute hidden under the weight read, array kept full.

Using torch-spyre's own cost-model constants — peak 98.3 TFLOPS DL16 (_PEAK_MACS_US_CORE = 1.536×106 MACs/µs/core × 32 cores) and HBM 204.8 GB/s — the machine balance (roofline knee) is 480 FLOP/byte. 5.5 Three things the gap is not

Not a relayout problem. The fused MLP compiles to 0 restickifies (confirmed in the pre-scheduling dump: exactly 8 ops, 3 batchmatmul + 5 pointwise, no restickify/stickify/reorient). The 2-kernel split's only real cost is a 16.78 MB HBM round-trip of the intermediate (~0.16 ms, minor). Utilization is the real gap.

Not a "200 ms/token" hardware ceiling. The ~36 ms decode spyre_ms device total is an isolated benchmark uploading all 402.7 MB of weights from the host every call (~33 ms of Memset/Memcpy). The actual matmul is ~2.8 ms; the upload amortizes to ~0 once weights stay resident, the way a real decoder runs. "200 ms/token" comes from multiplying the least-amortized metric by a layer count.

Not a batched-decode B× re-read bug. Batched decode lowers two ways. The shared-weight bmm_expand form reads the weights once and broadcasts across the batch — flat ~2.85 ms from B=1 to B=8. The real nn.Linear MLP lowers to this form, so production already amortizes the weight read. The distinct-weight bmm (which re-reads B× and grows linearly) only arises from explicit 3-D per-batch weights the model never produces — that is the kernel an old 12 ms profile measured, at ~B=2.5.

6. Closing the gap

Closing to parity is not a work-division / cost-model problem — the planner (PR #2407) already picks the right splits. It is kernel codegen, dominated by PT-array under-fill on the wide-N matmul. Most of it is deeptools-side (now in scope); the SiLU epilogue is the clearest Inductor-side win.

The levers, in priority order:

  1. Fill the PT array on the wide-N matmul (the dominant prefill term). Tile N=12800 onto the 8×8 array the way sendnn does, raising PT-Util from ~29.5% toward ~77.7%. Biggest single win; kernel codegen (deeptools/Inductor), not work division.
  2. SiLU in the matmul epilogue (~1.7 ms, prefill). Run the activation on the SFP unit concurrently with the matmul, as sendnn does, instead of a serial pass after. Mostly ours (Inductor fusion); needs the matmul SDSC to expose an epilogue hook. (This is a prefill win — at decode the intermediate is tiny and the FFN is already at parity, so there is no decode-MLP lever.)
  3. Fuse the FFN to one kernel. Removes the 16.78 MB intermediate HBM round-trip and enables (2). Inductor decides the fusion; deeptools executes it.
  4. Pipeline the chained matmuls (the beat lever). Let the down-projection's weight prefetch overlap the front matmul's compute. sendnn doesn't do this, so it's a genuine lead, not parity — but it only buys speedup after the array is filled (lever 1). Deeptools / SDSC overlap, same family as the on-chip ring producer→consumer work shown on attention.
Decode, restated. There is no decode-MLP lever. Re-measured on device, the decode FFN is at parity (tsp 2.84 / sendnn 2.76 ms = 1.03×) and at the weight-read floor; the bare GEMM is at parity too (0.83×). The standalone probe's 1.94× is per-call reload inflation, not compute (§3.2). None of levers 1–4 help decode — it is already at the memory ceiling. The only decode hygiene is keeping batched decode on the shared-weight bmm_expand form and weights resident; the win is entirely in prefill.

7. Methods & provenance

  • Authoritative build: torch-spyre 5838b3b (main), flex e0a343a0, deeptools e6534b3069, spyre-perf-suite e40013d (better_e2e_measures). Model: Granite 3.3 8B (40 layers, emb 4096, 8 GQA KV-heads, intermediate 12800). Run by the perf-suite owner; this is the "latest mains" reference.
  • Metric: kernel_ms = sum of compute device events (excludes Memset/Memcpy/host launch). Use ratios, not absolute ms, for the standalone op probes (~4× per-op overhead).
  • What is measured vs estimated: the whole-model prefill breakdown is tsp-only; there is no sendnn whole-model breakdown and no whole-model decode config in the report. The per-class sendnn split and the ~53/47 gap attribution are estimated via micro-probe ratios — robust in direction, ±10–15 points in magnitude. The whole-model decode 2.0× is carried from a prior run (unverified).
  • Bandwidth probes (143 GB/s read, 113 GB/s copy) and the shared-vs-distinct decode and SiLU-schedule measurements are independent on-device probes (single dedicated accelerator, runs strictly serial). The roofline ideals use torch-spyre's own cost-model constants.
