Folding the residual add into the matmul epilogue

A first-principles, device-measured account of the add_linear_mul fusion: what exactly was folded (the residual-connection +, not the GLU activation), why it collapses two SDSCs into one and erases an HBM round-trip, the subtle DeepTools ordering contract that the first cut got wrong, and what it buys on device. Companion to the MLP deep-dive and Closing the gap.

The one line
This PR folds the residual-connection add — the x + / h + that wraps each sub-block — into the preceding matmul (down_proj for the MLP, o_proj for attention) as a stridedadd SFP PSUM epilogue. The standalone [add] kernel and its HBM round-trip disappear: two SDSCs become one. It does not fuse the GLU's silu or the silu×mul gate — that is a separate, DeepTools-gated SwiGLU follow-up. On device this is −11% on o-proj and −3.2% on MLP-down, a near-constant ~0.09–0.11 ms saved per fold.

1. What was fused — and, just as importantly, what was not

The headline is easy to overclaim, so pin it down first. The name add_linear_mul is the Inductor-fused op label, and it is tempting to read it as "the GLU activation got fused into the matmul." That is not what happened. What was fused is the residual-connection add: the lone + that adds a sub-block's output back onto the residual stream. The GLU's own silu and its silu×mul gate are untouched and remain a separate, larger, DeepTools-gated prize.

Do not read this as "GLU activation fusion"
The residual + is a full-shape, elementwise add of the matmul output with the same-shape residual tensor. That is exactly what a stridedadd SFP epilogue does, and it is already in the matmul template's DDL menu (bmm.ddl stradd_op) and DDC whitelist — so an Inductor-side fold is correct end-to-end with no DeepTools change. The GLU gate (silu then a non-aligned multiply by the up-projection) is a different animal and needs DeepTools work; it is explicitly out of scope here.

2. The transformer block, and the two +s

Spelled out, one decoder layer is two residual-wrapped sub-blocks:

h = x + Attn(x)                                  # attention residual
y = h + MLP(h)                                   # MLP residual
  where MLP(h) = down_proj( silu(gate_proj(h)) * up_proj(h) )   # SwiGLU / GLU

Each + is a residual connection: it adds the sub-block's output back onto the tensor that entered it. Crucially, the value being added (Attn(x), MLP(h)) is the output of a matmul — o_proj closes attention, down_proj closes the MLP. So the residual add always sits immediately downstream of a matmul, reading that matmul's full-shape output and adding the skip tensor. That adjacency is the entire fusion opportunity.

The two GLU operations — silu(gate_proj(h)) and the * up_proj(h) multiply — live inside MLP(h), between the front projections and down_proj. They are not what this PR folds. The op label add_linear_mul names the Inductor group that the residual add lands in; the mul in that name is incidental graph fusion, not the SwiGLU gate getting absorbed into the matmul array.

3. First principles: why this is two SDSCs, and why folding makes it one

On Spyre a compiled op lowers to one or more SDSCs (Super-DSC kernel descriptors). Take the unfused residual: out = matmul(x, w) + r. Inductor schedules it as two nodes, and the backend emits two SDSCs:

  • [batchmatmul] — the matmul, writing its [M, N] product to an HBM intermediate.
  • [add] — a standalone SFP kernel that loads that intermediate back from HBM, loads the residual, adds, and stores the result.

The second kernel is pure tax. The matmul just produced its result in PSUM; the standalone add pushes it out to HBM and immediately streams it back in only to add a same-shape tensor to it. That round-trip is the cost the fold removes.

Fused, it is one SDSC carrying a 2-entry computeOp_ — [batchmatmul(pt), stridedadd(sfp)]. The stridedadd runs on the SFP unit against the matmul's PSUM output in place, reading the residual as an extra HBM input. The separate add kernel and its HBM round-trip of the matmul output simply never exist.

4. The DeepTools contract: the LAST entry is the output (an empirically-earned lesson)

This is the part the emit-only unit tests could not catch, and the cleanest first-principles lesson in the PR. A fused SDSC carries a list of labeled data-streams, labeledDs_ — one per tensor the kernel touches. DeepTools decides which one is the kernel's output purely by position:

  • The last labeledDs_ entry is the stored output. It is produced by the kernel; DeepTools does not attach a load for it.
  • Every earlier entry is an HBM-pinned input. DeepTools auto-attaches an HBM→LX load transfer and a producer for each.

So the residual, which is a genuine HBM input to the stridedadd, has to be spliced into the input section of the arg list — before the matmul output, which must stay last. Put it anywhere after the output and DeepTools reads it as a second "output," gives it no load and no producer, and the on-device compile aborts.

The lesson: emit-only tests passed straight through this bug
The 17 device-free unit tests verify the SDSC JSON we emit — that a 2-entry computeOp_ appears, that the residual is in the input section, that the output is the last labeledDs_. They all passed on the first (wrong-ordering) cut, because the emitted JSON was structurally valid. Only the on-device compile — DeepTools' DDC attaching producers by position — surfaced the fault. The first-principles takeaway: a fusion's correctness lives partly in a downstream contract you don't own (output = last by position), so an emit-shape test is necessary but not sufficient; the on-device compile is the real oracle. The fix itself is one line of intent — insert the residual before the output, not after.

5. The empirical numbers (device-measured)

Both residual sites measured on current main (53bbd76) through the real scheduler, unfused vs. fused, at prefill shapes (bs=1, sl=512, M=512):

matmul + residualshape [M, N, K]unfusedfusedΔsaved
o_proj + attn residual [512, 4096, 4096] 0.969 ms 0.862 ms −11% 0.107 ms
down_proj + MLP residual [512, 12800, 4096] 2.651 ms 2.565 ms −3.2% 0.086 ms
Honest framing — what this is, and is not
The saving is a roughly fixed ~0.09–0.11 ms per fold, with ~2 residual adds per layer, so on the order of ~1.5% of whole-prefill. That is a real, clean, shippable win and the proof-of-concept for matmul-epilogue fusion on Spyre — it lights up the stridedadd epilogue path Inductor-side with no DeepTools change. It is not the array-fill story (§6 of Closing the gap), and it is not the big GLU prize: folding the silu×mul gate into the matmul epilogue is the larger, DeepTools-gated next step.

6. Validation & the regression guard

  • Numerically correct on current main — the fused result matches the unfused matmul + residual reference.
  • 17 device-free unit tests (tests/inductor/test_fused_matmul_add.py) cover the fusion predicate, the spliced-input ordering, the output-stays-last invariant, the 2-entry computeOp_, the spec-tree fold to a single leaf, and the one-SDSC-JSON-per-pair emit.
  • Regression guard is green: a 1-D bias add stays unfused. The stridedadd epilogue requires a same-shape residual; a broadcast bias (shape [N]) is a biasadd, not a stridedadd, and folding it would leave its data-connect without a producer — so the predicate rejects any add whose non-matmul operand does not match the matmul output's element count.

The shape check is the single guard that keeps the fusion safe: both the Inductor-side can_fuse_matmul_residual_add gate and the backend _residual_add_fusion require residual.numel == matmul_out.numel, so only a true full-shape residual is ever folded.

Next GLU silu×mul gate → matmul epilogue DeepTools-gated (new epilogue contract) follow-up — the larger prize
Beyond fuse the whole FFN to one kernel; fill the wide-N array DeepTools / Inductor codegen see Closing the gap

The value of this PR beyond its own ~1.5% is that it establishes the pattern: a matmul SDSC can carry an SFP epilogue that operates on the PSUM output in place, scheduled Inductor-side, with the residual spliced into the input section so DeepTools attaches the right producer. The SwiGLU gate fusion reuses exactly this machinery — once DeepTools exposes a silu+aligned-mul epilogue, the same fold removes the FFN's intermediate round-trip too.
