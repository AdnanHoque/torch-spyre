What is actually happening in the MLP benchmark on Spyre

A first-principles reading of the mlp micro-benchmark in the spyre-perf-suite — what the op is, how it lowers onto the AIU 1.0 array, what each timing number means, and why the decode numbers look the way they do. Written to answer four questions: what's in the MLP, why are the numbers what they are, is 5–8 ms/MLP anomalous (→ ~200 ms/token?), and is that wall, spyre, or kernel time — plus a fifth (§10): why the wide-N [4096, 12800] matmul runs ~20% over sendnn, and whether that is a split-choice bug we can fix or a kernel/bandwidth wall.

1. The questions

The benchmark report shows, per MLP call, kernel times in the 2–8 ms range and a device-time (spyre_ms) around 20–35 ms. Three questions follow:

  1. What is the "mlp" op? One FFN block, or a whole layer, or a whole model?
  2. Is 5–8 ms per MLP anomalous? Multiplied by a layer count it implies something like ~200 ms/token, which would be far too slow for a decode step.
  3. Which clock is that? Wall-clock, total device time (spyre_ms), or just the matmul kernel time?
  4. And the wide-N matmul (§10): a separate flag — the standalone [1,512,4096]@[4096,12800] matmul (the MLP's front projection, measured alone) runs ~20% slower than sendnn. Is that the planner picking a bad split, or a kernel/bandwidth wall no split can beat?

The short answers, derived below: the op is a single SwiGLU FFN (two/three matmuls); the decode kernel time is not anomalous in the sense of "a bug," it is overhead-bound — at one token per step the 8×8 systolic array is essentially idle (PT-Util ~0.15%), so the time is dominated by fixed launch/stickify cost, not arithmetic; and the 200 ms/token figure comes from multiplying an isolated, per-call measurement by a layer count, which double-charges per-call data movement that amortizes away in a real autoregressive model.

2. The op: a SwiGLU FFN

Straight from the benchmark's op definition (benchmark.py, the mlp function):

def mlp(x, gate, up, down):
    gate_out   = x @ gate              # [M, 4096] @ [4096, 16384]
    up_out     = x @ up                # [M, 4096] @ [4096, 16384]
    swiglu_out = up_out * silu(gate_out)   # elementwise on [M, 16384]
    return swiglu_out @ down           # [M, 16384] @ [16384, 4096]

The tensor builder fixes the shapes: embedding dimension emb = 4096, and the intermediate ("hidden") dimension is 4 * emb = 16384. So the block is:

stepshapeMACs (per token, M=1)
gate = x@Wg[M,4096]@[4096,16384]67.1M
up = x@Wu[M,4096]@[4096,16384]67.1M
h = silu(gate)*upelementwise [M,16384]~0 (pointwise)
out = h@Wd[M,16384]@[16384,4096]67.1M
total—201.3M

This is the canonical Llama/Qwen-style SwiGLU MLP: two parallel "front" projections into a 4× wider space, a SiLU gate times the up-projection, then one "down" projection back to emb. It is one FFN block — not a layer (no attention, no norms), not a model (no layer stack). Everything below is the cost of that single block, run in isolation.

Two shapes are benchmarked:

  • Prefill: M = 512 tokens at once (a 512-token sequence). Total work ≈ 1.03×1011 MACs.
  • Decode: one token per step. The profiled decode case is a 4-token batch [4, 1, 4096] (4 independent sequences, one token each), i.e. M = 1 per sequence. Total work ≈ 8.05×108 MACs — about 128× less than prefill.

Hold onto that 128×: the decode kernel does ~128× less arithmetic, yet (we'll see) it takes about the same wall time as prefill. That gap is the whole story.

3. How it lowers on Spyre

The two stacks fuse the FFN differently, which matters when you read the per-kernel table.

torch-spyre (Inductor) splits the block into two compute kernels plus a Memset:

  • sdsc_fused_bmm_mul_silu_0 — the front half: gate-mm + up-mm + SiLU + the elementwise multiply, all fused into one SDSC kernel that emits the [M, 16384] intermediate.
  • sdsc_fused_bmm_1 — the down projection: [M,16384]@[16384,4096].
  • a Memset to zero an accumulator (sub-millisecond).

sendnn (the production backend) fuses the whole FFN into a single bmm kernel, plus host↔device Memcpy (HtoD / DtoH) and a Memset. So when comparing, remember: torch-spyre's "front + down" is two rows that you sum; sendnn's "bmm" is one row that already contains everything.

4. The three metrics, defined

The benchmark reports several clocks (see run_benchmark / write_stats in benchmark.py). Three matter, and conflating them is the source of the 200 ms/token confusion:

metricwhat it measuresincludes
kernel_ms Sum of the compute SDSC kernel durations (the bmm/sdsc_fused_* rows), excluding Memset and Memcpy. This is the actual PT-array work. matmul + fused pointwise on-device
PT-Util (%) How full the 8×8 systolic array is during the kernel — the fraction of the 64 multiply-accumulate cells doing useful work, averaged over the kernel. The smoking gun. array occupancy, not a time
spyre_ms Total device time for the call, summed over all device events: kernels plus per-call stickify, HBM staging, Memset, and (sendnn) host↔device Memcpy. everything the device did this call

There is also wall_clock_ms (Python perf_counter around the call, including the .to(device) / .cpu() transfers and dispatch — hundreds of ms here, dominated by host overhead and the per-call H2D/D2H of weights) and cpu_ms (self CPU time). Those are not the interesting signal for "is the accelerator slow."

Which clock is Jamie's table?
Jamie's per-kernel profile reports kernel_ms — "bmm kernel time only", broken down per kernel with PT-Util. It is not wall-clock and not spyre_ms. That distinction is the crux of §8.

The arresting fact: the decode front kernel (7.979 ms) is essentially the same as the prefill front kernel (7.109 ms) — despite doing 128× fewer tokens of work. If the kernel were compute-bound, decode would be ~128× faster. It isn't. The PT-Util column tells you why.

6. The PT-Util collapse

PT-Util goes from ~20–30% at prefill to ~0.15% at decode. That is a ~140× drop in array occupancy — the systolic array is essentially idle during the decode kernel.

Why does occupancy collapse? The matmul engine is an 8×8 systolic array that consumes 8 rows of M per pass (_PT_ROWS = 8 in work_division.py). It reaches full pipeline fill at roughly _TARGET_PT_PASSES = 8 passes — about 64 rows of M. The number of rows you feed it is M:

  • Prefill M = 512 → ~8 full passes per core (after an m-split) → the array fills.
  • Decode M = 1 → one row. A single row enters an 8×8 array, so at most ~1 of the array's rows is active and the other ~63 cells do nothing. The pipeline never fills; it spends its time in startup/drain.

7. Why decode is overhead-bound

If the array does no useful arithmetic, where does the ~3–8 ms per kernel go? Into fixed, per-kernel overhead that is independent of token count:

  • Kernel launch / dispatch — a fixed cost paid whether you matmul 1 row or 512.
  • Stickify of the [M, 16384] intermediate — the front kernel still materializes a 16384-wide intermediate (= 256 sticks wide). Laying that out in tiled stick format is a fixed layout cost dominated by the 16384 dimension, not by M.
  • K = 16384 reduction setup — the down projection reduces over K=16384 (256 sticks). The loop/accumulator setup over that long reduction axis is paid regardless of M.

This is why decode kernel time (7.979 ms front) ≈ prefill kernel time (7.109 ms front): the part that scales with tokens has shrunk to nothing, leaving only the part that doesn't. Decode is overhead-bound, not compute-bound.

The ~700× gap from the compute floor
Put a number on "the array is idle." One token's FFN is 2.013×108 MACs. The AIU's per-core peak is (98.304×1012 FLOPS / 2 / 32) = 1.536×106 MACs/µs/core (from _PEAK_MACS_US_CORE), so across 32 cores the floor is ~4.9×107 MACs/µs. The arithmetic lower bound for a 1-token FFN is therefore
2.013e8 / 4.9e7 ≈ 4 µs.
A single front matmul (gate or up, ~6.7e7 MACs) measures ~2.9 ms — about 700× the compute floor. 700× means: ~0.15% of the time is arithmetic, the rest is overhead. PT-Util and the 700× gap are two views of the same fact.

Contrast prefill: M=512 has 512× the MACs, so the compute floor is ~2.1 ms — now a meaningful fraction of the measured kernel time. That is exactly why prefill PT-Util is 20–40% (compute is a real share of the kernel) and decode PT-Util is ~0.15% (compute is a rounding error).

8. The "5–8 ms → 200 ms/token" claim, reconciled

The worry: a model has ~40 layers; 5 ms/MLP × 40 ≈ 200 ms/token, which would be a disaster. Two things are wrong with that multiplication.

8.1 It multiplies the wrong number

The 5–8 ms is an isolated single-op kernel measurement. Worse, the spyre_ms for the isolated call (~30 ms at decode for torch-spyre) includes per-call data movement that does not recur per layer in a real model:

  • In the benchmark, every call ships fresh weights and activations host↔device. sendnn's decode shows this nakedly: Memcpy HtoD = 10.601 ms, DtoH = 8.487 ms — the copies dwarf the 8.486 ms bmm. The benchmark is, at decode, more than half memcpy.
  • In a real autoregressive decoder, weights stay resident on-device across tokens and layers; activations stream through. You pay weight staging once, not per layer per token. The isolated benchmark charges it every call.

So if you must project per-token latency, the relevant quantity is the per-layer kernel time (~2–4 ms for the decode matmuls), not the per-call spyre_ms, and even that per-layer kernel time is inflated by the fixed overhead of §7 that a fused, resident-weight decoder would pay far less of (one launch and one stickify for a fused MLP, not two).

8.2 The latency claim hinges on metric + amortization

Whether "5–8 ms/MLP" is alarming depends entirely on which clock you propagate:

if you multiply…by 40 layersis it real?
spyre_ms isolated (~30 ms)~1200 msNo — double-charges per-call memcpy/stickify
kernel_ms sum (~12 ms)~480 msOverstated — per-layer launch+stickify amortizes when fused/resident
compute floor (~4µs/FFN)~0.16 msThe arithmetic the array actually needs to do

The truth is between the kernel-sum and the floor: real decode latency is set by how much of that fixed overhead you can amortize (resident weights, fused MLP, pipelined launches), not by the isolated-benchmark spyre_ms. The 200 ms/token figure is not a settled "too slow" — it is an artifact of multiplying the most pessimistic, least-amortized metric.

The honest one-liner
Decode kernel time is high because the array is idle (overhead-bound at M=1), not because the arithmetic is slow. The "200 ms/token" projection multiplies an isolated per-call cost (heavy on memcpy/stickify) that amortizes away once weights stay resident. It is a measurement / amortization question, not a proven hardware ceiling.

9. The LX-frac footnote

One backend knob colors any tsp-vs-sendnn comparison. DXP_LX_FRAC_AVAIL defaults to 0.2: only 20% of the LX scratchpad is offered to the matmul planner, with 80% reserved for the frontend. With less LX, the matmul has to chunk more and torch-spyre looks slower.

  • If that 80% reservation is genuinely idle during these matmuls (it is, in a single-op benchmark with nothing else running), then DXP_LX_FRAC_AVAIL = 1.0 is the fair comparison — and that is what Jamie's table uses (frac=1).
  • Crucially, the planner's split choice — the (m, n, k) it picks — is frac-independent. LX-frac changes how much chunking happens within a chosen split, not which split wins. So the underfill story above is unaffected by the frac setting; only the absolute tsp numbers move.

10. The wide-N matmul: why ~20% slower than sendnn

Separate from the decode story is a prefill flag: the standalone matmul [1, 512, 4096] @ [4096, 12800] — B=1, M=512, K=4096, N=12800 (= 200 sticks) — runs ~1.18–1.22× slower than sendnn in the field. This is the MLP's wide front projection measured in isolation. Two hypotheses: either the cost-model planner is picking a worse (m, n, k) split than it should (fixable in the planner), or the matmul kernel itself is slower regardless of split (a kernel/bandwidth gap). We settled it on device by forcing every legal split and timing each.

10.1 Why this matmul is bandwidth-bound, not compute-bound

Look at what moves. The weight is 4096 × 12800 × 2 B = 100 MB, which is ~86% of all bytes the kernel touches (the M=512 activation in/out are ~4 MB + ~13 MB). The arithmetic intensity (~5.4×1010 FLOPs over ~117 MB ≈ ~460 FLOP/byte) sits right at the AIU's machine balance, so the matmul is pinned against the HBM roofline knee: the floor is set by streaming that 100 MB weight through HBM (~204.8 GB/s → a ~0.6 ms bandwidth floor), not by array arithmetic. The decisive consequence: every split moves the same 100 MB weight. Work-division reshuffles which core does which output column, but the weight stream from HBM is invariant. You cannot divide your way out of a bandwidth wall.

10.2 The force-split experiment

On this node's card (serial, device-free for every run), I patched the work-division pass to force each legal full-32-core split and timed the matmul kernel. Two findings, one verdict
(1) No split spread. Every legal full-core split — pure-N (8,4), pure-M (32,1), and everything between — lands within ~2% of each other (run-to-run noise). (4,8) is not meaningfully faster than (8,4). The planner's tie-break between cohorts is irrelevant to throughput on this shape.

(2) The whole band sits ~25% over sendnn. Best tsp matmul kernel ≈ 1.18–1.20 ms vs sendnn's 0.962 ms → ~1.25×, matching the field slowdown. Even the fastest forced split stays ~20–25% above sendnn.

Verdict: it is a kernel/bandwidth gap, not a work-division problem. The slowdown lives in the matmul kernel itself (how well each stack streams the wide-N weight from HBM), exactly as the roofline argument in §10.1 predicts. The planner is choosing correctly; there is no split-choice win to capture here.

This is a useful negative result. An earlier paper analysis had hypothesized that ~half the gap was split choice — that (4,8) would beat the planner's pick because the cohort penalty is blind to weight rebroadcast. The device experiment refutes that: on hardware the splits are indistinguishable. The right place to chase the remaining 25% is the matmul kernel's weight-streaming codegen (a deeptools / kernel concern), not the inductor work-division planner.

A premise that didn't hold
On the worktree this ran against, the planner's default split for this shape is pure M=32, not the (8,4,1) an earlier write-up assumed (the two were built from different revisions). It does not change the verdict — (8,4,1) and (4,8,1) were forced explicitly and are no faster than the M=32 default — but it is worth flagging that "the planner picks (8,4,1) here" was itself incorrect.

Three things fall out, each tying back to a section above:

  1. The planner wins on every prefill shape, biggest where N is wide. qo_prefill 2.59× and mlp_prefill 2.07× over the heuristic. On qo_prefill the planner actually goes below sendnn (0.88×); the heuristic was leaving ~2.5× on the table by mis-splitting the same matmul.
  2. The mlp_prefill residual is the §10 wall. Even after a 2.07× win, ON is 1.20× sendnn — and §10's force-split proved that 20% is not recoverable by a different split (it's HBM weight streaming). So this row is "planner did its job, the rest is a kernel/bandwidth gap," not "planner picked wrong."
  3. Decode (bs4) barely moves, by design. At M=1 the array is underfilled regardless of split (the PT-Util collapse of §6–7), so OFF, ON, and sendnn all land within a few percent — there is almost no work-division lever at one token. ON still edges sendnn on qo_bs4 (0.96×) and mlp_bs4 (0.82×); the real decode lever is K-split to fill the idle array (§10 / takeaway 6), not N/M tuning.
What the A/B says in one line
The cost-model planner's value shows up at prefill, where M fills the array and the split actually matters: it roughly halves the heuristic's kernel time on the wide projections and reaches or beats sendnn on the square ones. Where it doesn't close the gap (mlp_prefill, 1.20×) the cause is the bandwidth wall of §10, not the split; where it can't move much (decode) the cause is the underfilled array of §6–7, not the split. Every residual traces to a kernel/hardware limit the planner is not responsible for.

12. Takeaways

  1. The op is one SwiGLU FFN block: gate & up project 4096→16384, SiLU-gate times up, down projects 16384→4096. Three matmuls, ~2×108 MACs per token.
  2. It lowers to 2 tsp kernels (front fused: gate+up+silu+mul; down) vs 1 sendnn bmm. Sum the two tsp rows to compare.
  3. Jamie's table is kernel time ("bmm kernel time only") — not wall, not spyre_ms.
  4. Decode is not anomalous, it is overhead-bound. At M=1 the 8×8 array runs ~1 row; PT-Util collapses from ~25% to ~0.15%; kernel time barely changes from prefill because the compute that should have shrunk 128× was never the bottleneck. Measured kernel is ~700× the ~4µs arithmetic floor.
  5. "200 ms/token" is a metric artifact. It multiplies an isolated per-call cost (in sendnn's decode, memcpy HtoD 10.6 ms > the bmm itself) that amortizes once weights stay resident across layers/tokens. Per-layer kernel time (~2–4 ms) is the right quantity, and even that is inflated by fixed launch/stickify overhead a fused resident decoder pays far less of.
  6. The real lever for decode is filling the idle array: K-split (split the K=16384 reduction across the otherwise-idle cores) is the only way to put work on more than ~1 row at M=1 — which is exactly where the cost-model's split-K decode path applies.
  7. LX-frac=1 is the fair comparison for an isolated benchmark; the split choice is frac-independent.
  8. The wide-N ~20% slowdown is a kernel/bandwidth gap, not a split problem (§10). Forcing every legal split of [1,512,4096]@[4096,12800] on device gives a ~2%-wide band all ~25% over sendnn; the matmul is HBM-bound streaming a 100 MB weight (~86% of bytes), so no split helps. Chase it in the matmul kernel's weight streaming, not in work division.
  9. The planner's value is concentrated at prefill (§11). Across the six matmul shapes it roughly halves the heuristic's kernel time on wide prefill projections (qo 2.59×, mlp 2.07×) and reaches/beats sendnn on the square ones; decode shapes barely move because the array is underfilled regardless of split. Every place it trails sendnn traces to §10 (bandwidth) or §6–7 (underfill), not to a wrong split.
