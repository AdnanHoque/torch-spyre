Why Layout Decides Matmul Speed: Fixing the Spyre Prefill Gap

A bs=1/sl=512 prefill matmul ran at 29.8% of peak. A pure-inductor change took it to 73% — matching the reference back end. The strange part: the FLOPs and the hardware never changed. What changed was how the op was presented to the weight-stationary array: its layout.

Wide-N MLP proj: 3.122 ms → 1.020 ms PT-Util: 29.8% → 73.0% vs sendnn 0.959 ms Diff: pure Python, 0 C++
The puzzle this document exists to answer

Same FLOPs. Same hardware. Same 32 cores. The matmul went from 29.8% to 73% utilization, and the diff was pure Python — not a single line of the back end's kernel generator changed.

How is that possible? The answer is one word — layout — and most people have never been taught what layout actually is, or why it governs how fast a matmul runs on a weight-stationary systolic array. So that is what the first half of this document builds, from zero, grounded in the Spyre microarchitecture paper and the tensor-layout spec.

Part I Building the intuition — how a weight-stationary array actually consumes a matmul, and why the same FLOPs can be fast or slow depending on how M is presented. Read this even if you already know what a matmul is.

1The weight-stationary array: tokens stream past resident weights

Before we can talk about layout, we have to look at the machine that consumes it. Everything about why layout matters falls out of one fact about Spyre's systolic array: it is weight-stationary. A tile of the weight matrix is loaded into the array and held resident, and the tokens of the activation are streamed past it.

Each Spyre core (the paper calls it a RaPiD corelet) has an 8×8 MPE array — 64 Mixed-Precision Engines wired as a 2D SIMD systolic grid that maps matrix multiplications. The paper is explicit about the dataflow direction and the on-chip plumbing that feeds it:

“…a 2D SIMD systolic array of 64 MPEs … to map matrix multiplications and convolutions. … L0 serves the rows of the MPE array, while L1 sits between L0/SFU/MPE columns and external memory.” Spyre microarchitecture paper, §3.3 Core Architecture

Here is how a matmul tile runs on it, in three steps:

  • A weight tile [K, N] is loaded into the array and held resident. The weights sit in the MPE cells (loaded into their register files); they do not move while the tile runs.
  • Tokens — the rows of the activation, the M dimension — stream through one after another. Each token is a K-vector that is fed across the array against the resident weights.
  • The N outputs accumulate as each token passes through, building up C[m, n] in the accumulator register file on the output edge.

The crucial consequence is about amortization. Loading the weight tile and filling the systolic pipeline (the fill/drain latency before the first and after the last result) is a fixed cost paid per resident-weight load. You only get that cost back by streaming many tokens past the resident weights before you reload. Stream few tokens per weight load and the fixed fill/drain + weight-load cost dominates — the array under-fills. Stream many tokens per weight load and the cost is amortized away — the array saturates.

The paper states this directly, and it is the load-bearing fact for the whole fix. tq is the prompt-token dimension — the M / row dimension of these matmuls — and over-splitting it (handing each core too few tokens per resident weight) wrecks utilization:

“Splitting tq beyond 8 results in under-utilization of the weight stationary dataflow within the corelets.” Spyre microarchitecture paper, §6.2 (prefill work assignment)

So the failure mode this entire document is about is not a scattered-memory feed. It is too few tokens streamed per resident weight: the array spends its time filling and draining a pipeline and reloading weights, instead of doing useful multiply-accumulates. Same total math, far fewer useful cycles.

Take this with you

A weight-stationary array's speed is decided not by how much math it has to do, but by how many tokens you stream past each resident-weight load. M is the streaming dimension. Split M too finely — hand each core too few tokens per weight load — and fill/drain + weight reload dominate. Hold that thought: layout is what decides whether M is even presented as a clean streaming dimension in the first place.

2Sticks: how Spyre actually stores a tensor

A tensor on Spyre is not a flat C array. Memory is organized into fixed sticks. To understand layout you first have to know what a stick is — and, critically, which matmul dimension each operand is sticked along.

A stick is the atomic, aligned unit of Spyre memory: 128 bytes = 64 fp16 elements, a 128-byte-aligned, 128-byte-contiguous block of tensor elements (tensors_and_layouts.md; BYTES_IN_STICK = 128 in spyre_tensor_impl.cpp). Hardware moves data a whole stick at a time — the size lines up with the natural granularity of transfers between device memory and the per-core LX/L1 scratchpad. A tensor's stick dimension is the logical dimension whose 64 elements sit physically contiguous and travel together.

Which logical dimension is the stick is not a free choice — for matmul the hardware fixes it per operand. The tensor-layout spec states the matmul stick convention precisely:

“Matrix multiply (C[m, n] = A[m, k] @ B[k, n]). A is sticked on the k dimension, while B and C are both sticked on the n dimension. B’s k dimension must be padded to full sticks …” tensors_and_layouts.md, §Per-operation stick and padding constraints

Read that carefully, because it is the single most-misunderstood fact about these matmuls:

  • The contiguous 64-element stick of the activation A[m,k] runs along k. A single token's k-vector is one stick.
  • The stick of the weight B[k,n] and the output C[m,n] runs along n.
  • M is never a stick dimension. Consecutive M-rows (consecutive tokens) are strided apart in memory by a whole stick (or more), not contiguous. M is the index over sticks, not the contents of one.

This dovetails with the weight-stationary dataflow from Section 1. Each token's k-stick is what gets fed across the array; the n outputs accumulate. The paper describes the same reuse structure for batched matmul:

“…in batch matrix multiplication (bmm), elements of the input tensor along its channel dimension (K) are reused along the output channel (N) dimension.” Spyre microarchitecture paper, §5.2 Segmented Work Assignment

3M, N, K — which is the stick, which is the stream

A matmul has three logical dimensions. Two stories run over them, and they are orthogonal: a stick story (which dim is the contiguous 64) and a streaming story (which dim flows past the resident weights). Keeping them apart is the whole game.

For C[M,N] = A[M,K] × B[K,N]:

  • M — output rows / tokens (e.g. the 512 sequence positions in a prefill). This is the streaming dimension — tokens flow past the resident weights, one after another. It is never a stick.
  • N — output columns (e.g. the 12800 features of a wide projection). The stick dimension of the weight and the output.
  • K — the contraction dim, summed away (e.g. the 4096 hidden size). The stick dimension of the activation; one token's k-vector is one stick fed across the array.

So there are two independent requirements to run efficiently on the 8×8 weight-stationary array:

  • The stick dimension is correct per operand — k on the input, n on the weight/output. This is the contiguity story from Section 2.
  • M is presented as a clean OUTER (streaming) dimension — a tidy outer loop over tokens, so the scheduler streams many of them past one resident-weight load. This is the amortization story from Section 1. If M is demoted to an inner position, the op is handed to the weight-stationary scheduler as a degenerate, short-streaming problem.
The rule, in one sentence

The array stays fed when the sticks are right (k in, n out — never M) and M is a first-class outer streaming dimension so many tokens flow past each resident-weight load. These are two different axes. Get the sticks right but demote M, and the array is fed too few tokens per weight load — it under-fills on the same math.

This is exactly why the prefill matmul ran slow. Not because the math was hard, not because the sticks were wrong — but because a decomposition demoted M out of its clean outer streaming position, so the op reached the weight-stationary scheduler as a degenerate bmm rather than a well-formed [M,K]×[K,N] matmul. We'll see precisely how in Part II. First, an analogy that makes the streaming/amortization intuition airtight.

4What “layout” means — and why it decides speed

Layout is the mapping from the logical [M,N,K] dimensions to their physical realization: which dim is the 64-stick (fixed for matmul: k in, n out), and which dim is the outer streaming iteration dim, in what order. Same numbers, same logical operation — but a different presentation to the weight-stationary scheduler.

The analogy: the inner kernel of a blocked GEMM

If you've ever hand-tuned a GEMM, you've already met this. In a blocked GEMM you load a tile of the weight matrix into fast on-chip storage and keep it resident, you hold an output tile accumulating in place, and you stream the activation rows through — reusing the resident weight tile against every row. The cost you are amortizing is the weight-tile load plus the pipeline fill. The more rows you stream per resident tile, the better that fixed cost is amortized and the closer you get to peak.

Now split the row dimension too finely across workers: each worker reloads the weight tile for only a handful of rows. The weight-load + fill cost is paid over and over for almost no useful work — amortization collapses, and throughput craters, even though the total arithmetic is identical. That is the failure mode here, in one sentence.

A note on labels, so the hardware claim stays exact: Spyre's array is weight-stationary — the weights are held resident in the MPE cells and the activations stream past. The GEMM analogy frames it as “output accumulating in place” only to convey the reuse pattern; the accumulator does sit on the output edge, but the resident thing in the silicon is the weight. Do not read this as an output-stationary machine.

The Spyre version is the same story, told in tokens and sticks:

Bad presentation — M demoted

The unit batch collapsed, leaving a degenerate bmm in which M is not a clean outer dimension. The scheduler sees a short-streaming problem: too few tokens per resident-weight load. The fixed weight-load + fill/drain cost dominates. Array under-fills.

like reloading a GEMM weight tile for only a handful of rows

Good presentation — M outer

M is a first-class outer streaming dimension; the sticks stay correct (k in, n out). The scheduler streams a long run of tokens past each resident-weight load. The fixed cost is amortized. Array saturates.

like streaming many rows through one resident GEMM weight tile

Notice the labels in the figure: [in, mb] on the left, [mb, in, x] on the right. Those are real dim_order values from the Spyre kernel descriptor, and the position of mb (M) within that list is literally the difference between demoted and outer. The stick entries (in = k, out = n) are unchanged; what moves is M's place in the iteration order. Part II shows exactly where those lists come from and why one had three entries and the other has four.

5Split vs. layout: two different knobs

This is the single most important distinction in the document, and the one most people conflate. Split and layout are independent. And note up front: both ultimately act on the same physical lever — how many tokens stream past each resident-weight load — which is why getting either wrong starves the array.

SPLIT — how much work each core gets

The 32 cores divide the matmul along three axes: m × n × k. A split of (m4, n8, k1) means 4 cores tile M, 8 tile N, 1 tiles K — 4×8×1 = 32 cores, each owning one rectangular chunk of the output. The m factor directly sets how many tokens each core streams per resident weight: m4 gives each core M/4 tokens, m8 only M/8. Over-split M and each core starves — the paper's “don't split tq beyond 8” rule.

LAYOUT — how the op is presented to the array

Before any split, the layout decides whether M is even presented as a clean outer streaming dimension with correct sticks (k in, n out), or whether a collapsed dim demotes M into a degenerate bmm. The layout decides whether the op reaches the weight-stationary scheduler as a well-formed streaming matmul at all.

Here is the punchline, and the whole reason this document exists: the layout fix changed how the op is presented, without changing the FLOPs or the kernel. Before the fix the unit batch collapsed, M was demoted, and the op reached the scheduler as a degenerate bmm — so even a sensible split could not stream enough tokens to fill the array. After the fix M is a clean outer streaming dimension, and (with the coupled planner edit in Section 11) the same hardware streams enough tokens per resident weight to saturate. Same FLOPs, different presentation, 29.8% → 73%.

6The op: a shared-weight unit BMM

The matmul that ran slow is an ordinary Transformer prefill projection. But by the time it reaches the layout machinery, a chain of decompositions has reshaped it into a peculiar form — a batched matmul whose batch is exactly 1. That “unit batch” is where the bug lives.

Take the wide MLP gate-up projection at batch-size 1, sequence-length 512:

# logical op: activation @ weight
torch.matmul([1, 512, 4096], [4096, 12800])

The activation is 3D [B=1, M=512, K=4096]; the weight is 2D [K=4096, N=12800] and shared across the batch (a normal Linear layer — every batch element multiplies the same weight, not a per-batch expert weight). Three things happen on the way down:

# 1. PyTorch flattens batch into rows
view(act, [B*M, K]) → aten.mm([512,4096], [4096,12800]) → view([B,M,N])

# 2. temp_passes.py:_unflatten_mm_to_bmm rewrites the view->mm->view back to a bmm,
#    expanding the 2D weight into a real 3D [1,K,N] tensor
bmm(act_3d[1,512,4096], expand(unsqueeze(weight))[1,4096,12800])

The unflatten exists because the Spyre backend tiles batched matmuls uniformly — it's easier to handle one bmm code path than two. The result is a bmm whose batch dimension is the literal integer 1 and whose weight is broadcast across that batch. That is the shared-weight unit BMM: 2D weight, statically-1 batch. The name pins down exactly the shape this fix is gated to, and distinguishes it from a genuine batched bmm (per-batch 3D weights, e.g. MoE experts), which this fix deliberately leaves alone.

Shared-weight unit BMM FIX TARGET

[1,M,K] @ [1,K,N] where the weight is the same broadcast matrix for the size-1 batch. The batch dim is an artifact of decomposition, not real parallel work.

Genuine batched BMM UNTOUCHED

[B,M,K] @ [B,K,N] with distinct per-batch weights (B>1, 3D weight). Real batch parallelism; the fix's recognizers return None for this.

7The bug: a collapsed unit dim demotes M

Now we can name the bug precisely. The unit-BMM shape was flattened, so torch-spyre stopped presenting the shared-weight prefill projection in the sendnn-like BMM layout. That led the back end (KG3) to a worse layout and split: too much splitting on M, not enough splitting on N, and weaker PT utilization. In the language of Part I, the collapse pushed M out of its clean-outer streaming position — but the load-bearing fact is the lost BMM layout, not a memory effect. The op reaches the weight-stationary scheduler as a degenerate, flattened matmul rather than the well-formed [M,K]×[K,N] BMM layout sendnn uses.

After the mm→bmm unflatten, the batch-of-1 dim has a device coordinate that is the constant integer 0 (Integer(0)) — it carries no free iteration symbol. The device-layout machinery (superdsc.py:_get_device_dim_order) only promotes dims that carry a symbol to iteration dims. With no symbol, the unit dim is treated as an inert, collapsed dim and dropped. The kernel descriptor then sees a 3-dim iteration space instead of four.

And the labels — which tell the layout machinery which dim is M, which is N, which is the stick — are assigned positionally. superdsc.py:_get_op_dim_labels returns the last ndim entries of MATMUL_DIM_LABELS from constants.py:

MATMUL_DIM_LABELS = ["ki","kj","y","x","mb","out","in"]

ndim = 3 → last 3 = ["mb", "out", "in"]      # BEFORE: degenerate bmm, M not a clean outer dim
ndim = 4 → last 4 = ["x", "mb", "out", "in"]  # AFTER: well-formed [M,K]x[K,N], M outer

With only three surviving symbols, they map positionally to [mb, out, in]. The consequence in device terms (this is the ground truth from the kernel descriptor):

TensorBEFORE device_sizeBEFORE dim_orderstick
input (act)[512, 64, 1, 64][in, mb]in (k)
output—[out, mb]out (n)

Read the dim_order: the sticks are correct (in = k on the input, out = n on the output), but M (mb) sits last rather than out front as a clean outer streaming dim. The op is presented to the weight-stationary scheduler as a degenerate bmm in which M is not a well-formed outer dimension — so the scheduler cannot stream a long run of tokens past each resident-weight load. With the stream cut short, the fixed weight-load + fill/drain cost dominates and the matmul under-fills at 29.8% (wide-N MLP) / 42.6% (QO-proj). This is the “M demoted” left-hand panel of Section 4, and exactly the paper's “under-utilization of the weight stationary dataflow.”

It was the layout first, then the split

An earlier investigation had noticed a "frozen (m4,n8,k1) split, dismissed as marginal." Tuning the split in isolation looked marginal because the op was being presented to the scheduler as a degenerate, flattened bmm underneath it — a good split cannot rescue an op the array can't stream. Restore the sendnn-like BMM layout first and the split lever comes alive (Section 11 shows it flipping from the unfixed (m8,n4) to the N-heavy (m4,n8)). This is the split-vs-layout distinction from Section 5, observed in the wild.

8The fix: re-introduce the dim, restore M as the outer stream

The fix does one conceptual thing: un-flatten the unit-BMM shape so torch-spyre presents the projection in the sendnn-like BMM layout again. Concretely it puts the unit batch dim back as a real iteration-space symbol and re-orders each tensor's device dims so M becomes a clean outer streaming dimension — handing KG3 the well-formed [M,K]×[K,N] BMM layout (which then splits less on M and more on N) instead of the flattened, degenerate form. Mechanically it's three coordinated parts — a marker that recognizes the shape, a thread that carries it through lowering, and the layout rewrite itself.

Part 1 — Recognize (commit afc9b57)

A marker key is added in constants.py: SHARED_WEIGHT_UNIT_BMM_CUSTOM_META_KEY = "_spyre_shared_weight_unit_bmm". temp_passes.py:_unflatten_mm_to_bmm tags the bmm node when there is exactly one batch dim and it is statically 1:

if len(batch_dims) == 1 and _is_static_one(batch_dims[0]):
    custom = dict(bmm_node.meta.get("custom") or {})
    custom[SHARED_WEIGHT_UNIT_BMM_CUSTOM_META_KEY] = {"batch_dim": 0}
    bmm_node.meta["custom"] = custom

_is_static_one does int(value) == 1 guarded against TypeError/ValueError, so it fires only for a concrete 1 — never for symbolic or >1 batch. The pre-existing pass_utils.py:copy_fx_custom_meta propagates meta["custom"] across the rewrite, so the marker survives.

Part 2 — Thread it through lowering (commit afc9b57)

lowering.py:lower_bmm reads the FX custom meta; if the marker is present, it builds an op_info dict and routes the reduction through ir.py:SpyreReduction.create(op_info=…) instead of the stock Reduction.create. (SpyreReduction is a pre-existing IR carrier whose whole job is to ferry op-specific info to codegen, since reductions bypass the virtualized ops API.) No marker → plain Reduction.create → unchanged behavior.

Part 3 — Re-introduce the dim and restore M as the outer stream (commit afc9b57)

This is the heart of the fix: spyre_kernel.py:_preserve_shared_weight_unit_bmm_dim. It replaces the constant-0 batch coordinate with a fresh iteration symbol _spyre_bmm_unit, reorders each tensor's device_size / device_coordinates / stride_map so the unit dim leads (the stick stays last — k on the input, n on the output, unchanged), and injects the unit dim into the iteration space as (S.One, 1). The iteration space becomes 4-dim, so the positional labels flip from [mb,out,in] to [x,mb,out,in] — restoring M to a clean outer position. Concretely:

Tensor / fieldBEFOREAFTER
input device_size[512, 64, 1, 64][1, 64, 512, 64]
iteration_space[mb, out, in][x, mb, out, in]
input dim_order[in, mb] M demoted[mb, in, x] M outer
output dim_order[out, mb] M demoted[mb, out, x] M outer
stickin (k) / out (n)in (k) / out (n) — unchanged

Read the dim_order rows top to bottom against Section 4's diagram: mb (M) moves from a demoted inner position to the front of the order. The stick is unchanged — still in (k) on the input, out (n) on the output. That single re-ordering restores M as a clean outer streaming dimension, so the weight-stationary scheduler can stream a long run of tokens past each resident weight. With the coupled planner edit (Section 11) keeping the split N-heavy on the now-4-dim space, the array fills → 73%.

The single difference that unlocks everything is the missing x:1 — the logical unit-BMM axis. Its size is 1, so dropping it cannot change the math, but it changes how the op is recognized and scheduled. Re-introducing it flips the layout from the flattened form to the sendnn-like BMM layout, and the split that follows from it is the clean one.

The arithmetic of the “after” shows why it is the right shape. The wide N projection is 12800 features; at 64 elements per stick that is 12800 / 64 = 200 output sticks. Splitting N across out=8 cores gives 200 / 8 = 25 sticks per core — versus 200 / 4 = 50 before. And M doubles per core: 512 / 4 = 128 rows streamed past each resident weight, versus 512 / 8 = 64 before. That is exactly the KB's MLP-prefill model made concrete: keep the weights stationary, stream more M rows past them (64 → 128 per core), and split the wide N projection cleanly into N tiles (25 sticks per core).

Why it was still correct before (the legality point)

Layout metadata is not the mathematical index formula by itself. The old SDSC was still a legal matmul — activation stick on in/K, weight & output stick on out/N — so the compiler still knew how to map every logical element to physical memory. It was just a poorer legal schedule/layout, not an incorrect one. That is why the buggy version produced byte-correct numbers (it complements the byte-identical-kernel point in Section 9). Size-1 axes are especially tricky here: mathematically they are removable, but for this back end they still carry useful scheduling/layout identity — drop the x:1 and you lose the BMM-layout identity that gets you the clean split.

Which recognizer fires for prefill

There is a second, size-based recognizer (spyre_kernel.py:_shared_weight_unit_bmm_info_from_sizes) as a fallback — but it requires the weight to be 2D. The real prefill path expands the weight to 3D [1,K,N], so the size fallback does not fire for prefill. The prefill win therefore depends on the FX marker surviving copy_fx_custom_meta. The 2D-only fallback covers a different entry point.

An honest note on the on-array claim

What is sourced: the weight-stationary dataflow, the matmul stick convention (k in, n out), and the “don't over-split tq” result (both paper + spec). What is device-verified: the before/after utilizations and the ablation. The precise on-array consequence of the dim-reorder — that M is restored as the outer streaming dimension and the scheduler therefore streams more tokens per resident-weight load — is the consistent reading of the kernel descriptor change, stated as such. We do not claim a PE-cell-level trace.

9The clincher: the kernel is byte-identical

If you remember one fact from Part II, make it this. The kernel descriptor's scalar fields are byte-for-byte identical before and after the fix. Only the iteration-space order, the dim_order, and the strides changed. Same kernel. Same FLOPs. Same hardware. We just presented the op so the weight-stationary array can stream it.

What did NOT change (byte-identical)

  • opfunc = batchmatmul
  • num_cores = 32
  • num_inputs = 2
  • data_format (e.g. SEN169_FP16)
  • the stick dims: in (k) / out (n)
  • the FLOPs, the dtype, the hardware

What DID change (presentation only)

  • iteration space: [mb,out,in] → [x,mb,out,in]
  • input dim_order: [in,mb] → [mb,in,x]
  • output dim_order: [out,mb] → [mb,out,x]
  • the strides that follow from the new order
  • M's role: demoted (inner) → clean outer stream

This is the entire lesson of the document, made concrete: a matmul's speed on a weight-stationary array is governed by how the op is presented — whether M reaches the scheduler as a clean outer streaming dimension — not just by how the work is divided. The same op, the same FLOPs, the same silicon went from 29.8% to 73% because M was finally presented so the array could be fed enough tokens per resident weight.

Verified, and INDUCTOR-only

Reproduced on our local /opt deeptools (the older one, not codex's newer back end) at 73.7% / 72.7%. All 10 fix tests plus 117 test_coarse_tiling tests pass. Because the speedup needs no back-end change — the layout is the back end's input, not its capability — this is an inductor-only fix, not a deeptools-version effect.

Part III Context and caveats — the coupled planner edit, two misconceptions corrected, and an honest map of what this does and does not fix.

11The planner edit — also load-bearing

A second change rides along on commit 6a383d8. An ablation — revert only this, keep the layout fix — shows it is not secondary: it is worth up to +25 utilization points on the wide projection. Both fixes are needed, and both act on the same physical lever — tokens streamed per resident weight.

The work-division planner (work_division.py:_matmul_split_cost) had a symmetric tie-break term, target_m_us, a V-shaped penalty that punished fewer-M-core splits exactly as hard as over-splitting M. For wide projections this "incorrectly prefers M-parallelism over N-parallelism" (the commit's own words). The edit rewrites it into a one-sided floor penalty — punish only M-splits that leave each core with too few tokens to amortize the resident weight:

# NEW (6a383d8) — penalize only PT under-fill (too few tokens per core)
if pt_passes >= _TARGET_PT_PASSES:        # per-core M tile already streams enough tokens
    target_m_us = 0.0
else:
    target_m_us = math.log2(_TARGET_PT_PASSES / pt_passes) * _TARGET_M_PENALTY_US

This is literally the paper's “don't over-split tq” point in cost-model form. A finer M-split (m8) gives each core fewer tokens per resident weight; a coarser one (m4) gives more. Penalizing only the under-fill direction lets the planner stop at m4 instead of drifting to the starving m8 — more tokens per core, better weight-stationary amortization.

The split, before and after (authoritative)

Jamie's probe (jamie_unit_bmm_probe.md, base a4c2449) gives the canonical before/after for the wide-N projection's split. The unfixed, flattened form splits too much on M and too little on N; the full fix un-flattens to the sendnn-like BMM layout and the split inverts — fewer M-tiles, more N-tiles:

configsplit per core (M rows / N sticks)
BEFORE (unfixed, flattened)(m8,n4)64 rows / 50 sticks
AFTER (full fix)(m4,n8)128 rows / 25 sticks

So the two fixes are coupled. The layout fix un-flattens the op into the sendnn-like BMM layout and presents M as a clean outer stream; on that restored 4-D space, the old cost model still picks the M-heavy split, and the planner edit is what flips it to the N-heavy (m4,n8) the wide projection actually wants — 128 M rows streamed per core (vs 64) and the wide N projection split cleanly into 25 sticks per core (vs 50). Our own ablation independently confirmed the cost-model edit is load-bearing: on our build, the new layout with the old cost model reached only ~47% PT-Util, versus ~73% with the edit. Both the layout restoration and the N-heavy split are required for 73%. (An earlier static reading claimed the planner "declines" for this path — the ablation refutes it; the planner engages and the split visibly flips.)

The exact unfixed split is base-dependent — a hand-port on an older base (53bbd76) showed a different unfixed split — but the direction (the full fix lands on the N-heavy (m4,n8), and both levers are needed) holds. Jamie's probe on the fix's actual base (a4c2449) is the authoritative reading.

12Correcting two misconceptions

✗ Misconception A: the “hard chunk-cap” (refuted)

"The ~29.5% prefill matmul plateau is a hard DeepTools chunk-optimizer cap — an array-fill ceiling unreachable from inductor, with no inductor lever. The only inductor observation was a frozen (m4,n8,k1) split, dismissed as marginal."

✓ What was actually true (A)

The plateau was an inductor layout + split artifact, not a hardware or back-end ceiling. The size-1 batch dim collapsed, the iteration space dropped to a degenerate 3-dim [mb,out,in] space, M was demoted out of its outer streaming position, and the weight-stationary scheduler could not stream enough tokens per resident weight. An inductor-only fix reaching 73% on the same --bundle back end refutes the “no inductor lever” framing. DeepTools was faithfully executing a badly-presented matmul; the same compiler builds a fast kernel the moment inductor hands it a well-formed, M-outer layout (plus the coupled planner split).

✗ Misconception B: “M is the contiguous / coalesced dimension” (physically wrong)

An earlier writeup pictured M as the memory-contiguous dimension — consecutive output rows as “one contiguous run, the next row is the next stick over,” with the slowdown framed as an uncoalesced / scattered-memory read (a GPU coalescing analogy), and the systolic array drawn as being filled by streaming M-rows of contiguous memory. This mental model is physically wrong.

✓ What was actually true (B)

The contiguous 64-element stick is along k for the activation and along n for the weight and output (tensors_and_layouts.md). M is never a stick — consecutive M-rows are strided apart, not contiguous, and they were strided apart correctly both before and after the fix. The array is weight-stationary: a weight tile is held resident and tokens (the M / tq dimension) stream past it. So M matters as the streaming / temporal dimension over a stationary weight, not as a memory-contiguous one. The stick story (k in, n out) is orthogonal to the M story; the old doc collapsed these two different axes into one wrong picture. The real slowdown was M demoted out of the outer streaming position (too few tokens per resident-weight load), not a scattered read.

13What this does NOT fix

The fix is tightly gated (2D weight, batch dim statically 1). It deliberately leaves several adjacent cases untouched, and it is honest about its blast radius.

Decode matmuls (M=1 / M=4)

Unchanged at ~0.8% util — and expected. Decode is weight-bound: M is so tiny there are too few tokens to stream regardless of layout, so restoring M as an outer dim can't help. A different regime (data movement, not array-fill).

Genuine batched BMM (true 3D weights)

The [4,1,4096] per-batch-weight op stays at ~0.07% util. The recognizers return None for a 3D weight — this fix targets the shared-weight (2D, unit-batch) case only. MoE-style experts are a separate lever.

Out of scope: the MLP LX-overflow crash

This fix is about array-fill on the matmul. The separate prefill-MLP issue — the on-chip LX-residency / overflow behavior governed by the DeepTools LX allocator (a compile-time crash, not a utilization number) — is a different problem and is not touched here.

One-line reference: the attention fix

A separate commit (824ad4b on the sibling branch pr-mlp-fix) bypasses the SDPA fake-tensor setup that mis-fires on the Spyre device. It is an attention correctness fix, unrelated to this matmul layout work, and is mentioned only so it isn't confused with it.
