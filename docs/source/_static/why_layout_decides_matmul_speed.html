Why Layout Decides Matmul Speed: Fixing the Spyre Prefill Gap

A bs=1/sl=512 prefill matmul ran at 29.8% of peak. A pure-inductor change took it to 73% — matching the reference back end. The strange part: the work was split across the 32 cores exactly the same way before and after. Only the layout changed.

Wide-N MLP proj: 3.122 ms → 1.020 ms PT-Util: 29.8% → 73.0% vs sendnn 0.959 ms Diff: pure Python, 0 C++
The puzzle this document exists to answer

Same FLOPs. Same hardware. Same 32 cores. The same work-division split (m4 × n8 × k1) before and after. And yet the matmul went from 29.8% to 73% utilization.

How is that possible? The answer is one word — layout — and most people have never been taught what layout actually is or why it governs how fast a matmul runs. So that is what the first half of this document builds, from zero.

Part I Building the intuition — what layout is, and why the same split can be fast or slow depending on it. Read this even if you already know what a matmul is.

1The systolic array: a machine that must be fed

Before we can talk about layout, we have to look at the machine that consumes the layout. Everything about why layout matters falls out of one fact: a systolic array is a streaming engine, and a streaming engine is only fast if you never stop feeding it.

Each Spyre core has an 8×8 PT systolic array — a grid of 64 multiply-accumulate (MAC) cells. It does not compute a matmul the way a CPU does (load operands, multiply, store). Instead, data physically marches through the grid, one step per clock cycle:

  • One operand flows in as a continuous stream — row by row, cycle by cycle, entering the left edge and rippling rightward.
  • The other operand sits stationary in the cells.
  • As each value passes a cell, it is multiplied and added into a running partial sum. Partial sums accumulate as the data wave travels across the grid.

The crucial consequence: the array has a fixed pipeline that must fill and drain. The first result doesn't emerge until the data wave has crossed the whole grid (fill); the last results trickle out after the input ends (drain). Fill and drain are pure overhead — cycles where the array is not yet, or no longer, doing full useful work. They are only amortized if you push a long, uninterrupted stream through afterward.

And here is the failure mode that this entire document is about. If, on any cycle, the next value the array needs isn't ready — because it is somewhere else in memory, out of order, waiting to be gathered — the array stalls. That stall is called a bubble: a cycle where MAC cells spin with no valid data. Every bubble is throughput thrown on the floor. A systolic array fed a clean stream runs near 100%; the same array fed a stuttering, scattered stream runs at a fraction of that — doing the exact same math.

Take this with you

A systolic array's speed is decided not by how much math it has to do, but by whether it can be fed a long, in-order, contiguous stream with no bubbles. Hold that thought — layout is precisely the thing that determines whether such a stream exists.

2Sticks: how Spyre actually stores a tensor

A tensor on Spyre is not a flat C array. Memory is organized into fixed sticks, and the array is fed stick-by-stick. To understand layout you first have to know what a stick is.

A stick is the atomic, aligned unit of Spyre memory: 128 bytes = 64 fp16 elements. Hardware moves, addresses, and feeds data in whole sticks. A tensor is tiled into sticks; its innermost 64-element dimension is the stick dimension — the 64 values that sit physically contiguous and travel together.

Think of it like cache lines, but mandatory and load-bearing: the hardware can only efficiently grab data a stick at a time. A value's neighbors within its stick are free to reach; a value's neighbors in a different stick cost a separate fetch. So which logical dimension you choose to be the 64-wide stick, and how the other dimensions are ordered around it, decides whether the data the array wants next is the next 64 contiguous bytes (good) or 64 bytes scattered across far-apart sticks (bad — the bubble machine from Section 1).

3M, N, K and the one job of a layout

A matmul has three logical dimensions. Mapping them onto sticks and stream order is the whole game. Let's name them, then state the rule the hardware imposes.

For C[M,N] = A[M,K] × B[K,N]:

  • M — output rows (e.g. the 512 sequence positions in a prefill).
  • N — output columns (e.g. the 12800 features of a wide projection).
  • K — the contraction dim, summed away (e.g. the 4096 hidden size).

To run efficiently on the 8×8 array, the data has to be arranged so that:

  • One dimension is the 64-wide stick — the contiguous unit that streams through cells.
  • M is presented as a clean OUTER iteration dimension — the array processes one M-tile, then the next, then the next, marching down the rows. If M is a tidy outer loop, the hardware knows exactly where the next row of work lives: it's the next stick over. It streams row after row at full rate.
The rule, in one sentence

The array stays fed when M is a first-class outer dimension (so rows stream one after another) and the stick dimension is a clean contiguous 64-wide run. Break either — bury M among the inner dims, or scatter the stick — and the next value isn't where the array expects it, so you get bubbles. Same math, fewer useful cycles.

This is exactly why the prefill matmul ran slow. Not because the math was hard, not because the work was badly divided among cores — but because the data got arranged so that M was not a clean outer dimension. We'll see precisely how in Part II. First, let's make the intuition airtight with an analogy you may already carry.

4What “layout” means — and why it decides speed

Layout is the mapping from the logical [M,N,K] dimensions to their physical realization: which dim is the 64-stick, which dims are outer iteration dims and in what order, and the strides that follow from that order. Same numbers, same logical operation — but a different physical arrangement in memory and a different stream order into the array.

The analogy: coalesced vs. uncoalesced access

If you've written GPU kernels, you've met this exact phenomenon under a different name. On a GPU, when a warp of threads reads memory, the hardware is fast only if the threads touch contiguous addresses — a coalesced access, served as one wide transaction. If the threads touch scattered addresses, the read uncoalesces into many small transactions and the compute units stall waiting on the gather. Identical arithmetic; the access pattern alone makes it fast or slow.

Or, if you've ever fed a GEMM library the wrong leading dimension — passed a row-major matrix to a kernel expecting column-major, or transposed an operand it wanted untransposed — you've watched a matmul crater for no reason except that the rows it wanted next were strided far apart in memory. The FLOPs didn't change. The layout did.

The systolic-array version is the same story, told in sticks:

Bad layout — M buried

M is interleaved with the inner stick dim, not a clean outer loop. The row of work the array needs next is not the next stick in memory — it's scattered. The feed stutters while the hardware gathers. Bubbles. Under-fill.

like an uncoalesced GPU read / the wrong leading dimension

Good layout — M outer

M is a first-class outer dimension; the stick is a clean contiguous 64. The next row of work is the next stick over. The array streams row after row, no gathers, no stalls. Saturated. Full util.

like a coalesced GPU read / the matching leading dimension

Notice the labels in the figure: [in, mb] on the left, [mb, in, x] on the right. Those are real dim_order values from the Spyre kernel descriptor, and the position of mb (M) within that list is literally the difference between buried and outer. Part II shows exactly where those lists come from and why one had three entries and the other has four.

5Split vs. layout: two different knobs

This is the single most important distinction in the document, and it's the one most people conflate. Split and layout are independent. A perfect split with a broken layout still starves the array.

SPLIT — how much work each core gets

The 32 cores divide the matmul along three axes: m × n × k. A split of (m4, n8, k1) means 4 cores tile M, 8 tile N, 1 tiles K — 4×8×1 = 32 cores, each owning one rectangular chunk of the output. The split decides the size and shape of each core's sub-problem.

LAYOUT — how that chunk is arranged for streaming

Given a core's chunk, the layout decides whether its data is arranged so the array can stream it — M outer, stick contiguous — or whether M is buried and the feed stutters. The layout decides whether each core's work actually fills its array.

Here is the punchline, and the whole reason this document exists: the fix changed only the layout. The split was (m4, n8, k1) before the fix and (m4, n8, k1) after it. Each core received the identical chunk of work. But before, that chunk was laid out with M buried, so the array under-filled; after, the same chunk was laid out with M outer, so the array saturated. Same split, different layout, 29.8% → 73%.

6The op: a shared-weight unit BMM

The matmul that ran slow is an ordinary Transformer prefill projection. But by the time it reaches the layout machinery, a chain of decompositions has reshaped it into a peculiar form — a batched matmul whose batch is exactly 1. That “unit batch” is where the bug lives.

Take the wide MLP gate-up projection at batch-size 1, sequence-length 512:

# logical op: activation @ weight
torch.matmul([1, 512, 4096], [4096, 12800])

The activation is 3D [B=1, M=512, K=4096]; the weight is 2D [K=4096, N=12800] and shared across the batch (a normal Linear layer — every batch element multiplies the same weight, not a per-batch expert weight). Three things happen on the way down:

# 1. PyTorch flattens batch into rows
view(act, [B*M, K]) → aten.mm([512,4096], [4096,12800]) → view([B,M,N])

# 2. temp_passes.py:_unflatten_mm_to_bmm rewrites the view->mm->view back to a bmm,
#    expanding the 2D weight into a real 3D [1,K,N] tensor
bmm(act_3d[1,512,4096], expand(unsqueeze(weight))[1,4096,12800])

The unflatten exists because the Spyre backend tiles batched matmuls uniformly — it's easier to handle one bmm code path than two. The result is a bmm whose batch dimension is the literal integer 1 and whose weight is broadcast across that batch. That is the shared-weight unit BMM: 2D weight, statically-1 batch. The name pins down exactly the shape this fix is gated to, and distinguishes it from a genuine batched bmm (per-batch 3D weights, e.g. MoE experts), which this fix deliberately leaves alone.

Shared-weight unit BMM FIX TARGET

[1,M,K] @ [1,K,N] where the weight is the same broadcast matrix for the size-1 batch. The batch dim is an artifact of decomposition, not real parallel work.

Genuine batched BMM UNTOUCHED

[B,M,K] @ [B,K,N] with distinct per-batch weights (B>1, 3D weight). Real batch parallelism; the fix's recognizers return None for this.

7The bug: a collapsed unit dim buries M

Now we can name the bug precisely in the language of Part I: the size-1 batch dim got silently dropped, which knocked the iteration space down to three dimensions and pushed M out of its clean-outer position — the exact "M buried" failure mode.

After the mm→bmm unflatten, the batch-of-1 dim has a device coordinate that is the constant integer 0 (Integer(0)) — it carries no free iteration symbol. The device-layout machinery (superdsc.py:_get_device_dim_order) only promotes dims that carry a symbol to iteration dims. With no symbol, the unit dim is treated as an inert, collapsed dim and dropped. The kernel descriptor then sees a 3-dim iteration space instead of four.

And the labels — which tell the layout machinery which dim is M, which is N, which is the stick — are assigned positionally. superdsc.py:_get_op_dim_labels returns the last ndim entries of MATMUL_DIM_LABELS from constants.py:

MATMUL_DIM_LABELS = ["ki","kj","y","x","mb","out","in"]

ndim = 3 → last 3 = ["mb", "out", "in"]      # BEFORE: degenerate 2D-mm layout, M buried
ndim = 4 → last 4 = ["x", "mb", "out", "in"]  # AFTER: sendnn-like unit layout, M outer

With only three surviving symbols, they map positionally to [mb, out, in]. The consequence in device terms (this is the ground truth from the kernel descriptor):

TensorBEFORE device_sizeBEFORE dim_orderstick
input (act)[512, 64, 1, 64][in, mb]in
output—[out, mb]out

Read the dim_order: M (mb) sits second, adjacent to the 64-element stick, rather than out front as a clean outer row dim. This is the "M buried next to the stick" situation from Section 4 — exactly the left-hand panel of that diagram. The array cannot stream a clean run of M-rows; consecutive rows are strided apart, every row is a gather, the pipeline bubbles, and the matmul under-fills at 29.8% (wide-N MLP) / 42.6% (QO-proj).

It was the layout all along, not the split

An earlier investigation had noticed a "frozen (m4,n8,k1) split, dismissed as marginal." That split was already correct — it only looked marginal because the layout was broken underneath it. A good split feeding a buried-M layout still starves the array. This is the split-vs-layout distinction from Section 5, observed in the wild.

8The fix: re-introduce the dim, re-order the layout

The fix does one conceptual thing: put the unit batch dim back as a real iteration-space symbol, and re-order each tensor's device dims so M becomes a clean outer dimension again. Mechanically it's three coordinated parts — a marker that recognizes the shape, a thread that carries it through lowering, and the layout rewrite itself.

Part 1 — Recognize (commit afc9b57)

A marker key is added in constants.py: SHARED_WEIGHT_UNIT_BMM_CUSTOM_META_KEY = "_spyre_shared_weight_unit_bmm". temp_passes.py:_unflatten_mm_to_bmm tags the bmm node when there is exactly one batch dim and it is statically 1:

if len(batch_dims) == 1 and _is_static_one(batch_dims[0]):
    custom = dict(bmm_node.meta.get("custom") or {})
    custom[SHARED_WEIGHT_UNIT_BMM_CUSTOM_META_KEY] = {"batch_dim": 0}
    bmm_node.meta["custom"] = custom

_is_static_one does int(value) == 1 guarded against TypeError/ValueError, so it fires only for a concrete 1 — never for symbolic or >1 batch. The pre-existing pass_utils.py:copy_fx_custom_meta propagates meta["custom"] across the rewrite, so the marker survives.

Part 2 — Thread it through lowering (commit afc9b57)

lowering.py:lower_bmm reads the FX custom meta; if the marker is present, it builds an op_info dict and routes the reduction through ir.py:SpyreReduction.create(op_info=…) instead of the stock Reduction.create. (SpyreReduction is a pre-existing IR carrier whose whole job is to ferry op-specific info to codegen, since reductions bypass the virtualized ops API.) No marker → plain Reduction.create → unchanged behavior.

Part 3 — Re-introduce the dim and re-order the layout (commit afc9b57)

This is the heart of the fix: spyre_kernel.py:_preserve_shared_weight_unit_bmm_dim. It replaces the constant-0 batch coordinate with a fresh iteration symbol _spyre_bmm_unit, reorders each tensor's device_size / device_coordinates / stride_map so the unit dim leads (stick stays last), and injects the unit dim into the iteration space as (S.One, 1). The iteration space becomes 4-dim, so the positional labels flip from [mb,out,in] to [x,mb,out,in] — the sendnn-like unit layout. Concretely:

Tensor / fieldBEFOREAFTER
input device_size[512, 64, 1, 64][1, 64, 512, 64]
iteration_space[mb, out, in][x, mb, out, in]
input dim_order[in, mb] M buried[mb, in, x] M outer
output dim_order[out, mb] M buried[mb, out, x] M outer
stickin / outin / out (unchanged)

Read the dim_order rows top to bottom against Section 4's diagram: mb (M) moves from the buried position next to the stick to the front of the order. The stick stays in (input) / out (output). That single re-ordering is what turns the strided, gather-every-row feed into a contiguous M-stream the array can saturate. The same (m4×n8) split now fills the array → 73%.

Which recognizer fires for prefill

There is a second, size-based recognizer (spyre_kernel.py:_shared_weight_unit_bmm_info_from_sizes) as a fallback — but it requires the weight to be 2D. The real prefill path expands the weight to 3D [1,K,N], so the size fallback does not fire for prefill. The prefill win therefore depends on the FX marker surviving copy_fx_custom_meta. The 2D-only fallback covers a different entry point.

9The clincher: the kernel is byte-identical

If you remember one fact from Part II, make it this. The kernel descriptor's scalar fields are byte-for-byte identical before and after the fix. Only the iteration-space order, the dim_order, and the strides changed. Same kernel. Same split. Same hardware. We just told it to lay the data out so the array can stream it.

What did NOT change (byte-identical)

  • opfunc = batchmatmul
  • num_cores = 32
  • num_inputs = 2
  • data_format (e.g. SEN169_FP16)
  • the split: (m4, n8, k1)
  • the FLOPs, the dtype, the hardware

What DID change (layout only)

  • iteration space: [mb,out,in] → [x,mb,out,in]
  • input dim_order: [in,mb] → [mb,in,x]
  • output dim_order: [out,mb] → [mb,out,x]
  • the strides that follow from the new order
  • M's position: buried → outer

This is the entire lesson of the document, made concrete: a matmul's speed on a systolic array is governed by layout, not just by how the work is divided. The same op, the same split, the same silicon went from 29.8% to 73% because the data was finally arranged so the array could be fed.

Verified, and INDUCTOR-only

Reproduced on our local /opt deeptools (the older one, not codex's newer back end) at 73.7% / 72.7%. All 10 fix tests plus 117 test_coarse_tiling tests pass. Because the speedup needs no back-end change — the layout is the back end's input, not its capability — this is an inductor-only fix, not a deeptools-version effect.

Part III Context and caveats — the coupled planner edit, a misconception corrected, and an honest map of what this does and does not fix.

11The planner edit — also load-bearing

A second change rides along on commit 6a383d8. An ablation — revert only this, keep the layout fix — shows it is not secondary: it is worth up to +25 utilization points on the wide projection. Both fixes are needed.

The work-division planner (work_division.py:_matmul_split_cost) had a symmetric tie-break term, target_m_us, a V-shaped penalty that punished fewer-M-core splits exactly as hard as over-splitting M. For wide projections this "incorrectly prefers M-parallelism over N-parallelism" (the commit's own words). The edit rewrites it into a one-sided floor penalty — punish only M-splits that leave the per-core tile too short to fill the PT pipeline:

# NEW (6a383d8) — penalize only PT under-fill
if pt_passes >= _TARGET_PT_PASSES:        # per-core M tile already fills PT
    target_m_us = 0.0
else:
    target_m_us = math.log2(_TARGET_PT_PASSES / pt_passes) * _TARGET_M_PENALTY_US
The ablation: both levers are necessary

Here is the subtlety. Once the layout fix re-introduces the unit dim, the iteration space is 4-D and the planner engages. On that new space the old cost model picks the M-heavy (m8,n4) split; the edit flips it to the N-heavy (m4,n8) the wide projection actually wants. Reverting only the planner edit (keeping the layout fix) measures its weight directly — device-verified:

configsplit QO‑proj utilwide‑N util
no fix(m4,n8)42.6%29.8%
layout only (planner reverted)(m8,n4)62.3%47.4%
full fix(m4,n8)73.7%72.7%

So the two fixes are coupled: the layout fix makes the data streamable and restructures the iteration space; the planner edit then keeps the split N-heavy on that new space (otherwise the old cost model drifts to the worse (m8,n4) and stalls at 47%). Layout alone gets 47–62%; the planner edit adds the rest. Both are required for 73%. (An earlier static reading claimed the planner "declines" for this path — the ablation refutes it; the planner engages and the split visibly flips.)

12Correcting the “chunk-cap” misconception

✗ The old story (refuted)

"The ~29.5% prefill matmul plateau is a hard DeepTools chunk-optimizer cap — an array-fill ceiling unreachable from inductor, with no inductor lever. The only inductor observation was a frozen (m4,n8,k1) split, dismissed as marginal."

✓ What was actually true

The plateau was an inductor layout artifact, not a hardware or back-end ceiling. The size-1 batch dim collapsed, the iteration space dropped to a degenerate 3-dim [mb,out,in] space, M got buried next to the stick, and the array couldn't be streamed. An inductor-only fix reaching 73% on the same --bundle back end refutes the "no inductor lever" framing. DeepTools was faithfully executing a badly-laid-out matmul; the same compiler builds a fast kernel the moment inductor hands it a streamable layout.

13What this does NOT fix

The fix is tightly gated (2D weight, batch dim statically 1). It deliberately leaves several adjacent cases untouched, and it is honest about its blast radius.

Decode matmuls (M=1 / M=4)

Unchanged at ~0.8% util — and expected. Decode is weight-bound: M is so tiny there's nothing to stream regardless of layout, so a layout rewrite can't help. A different regime (data movement, not array-fill).

Genuine batched BMM (true 3D weights)

The [4,1,4096] per-batch-weight op stays at ~0.07% util. The recognizers return None for a 3D weight — this fix targets the shared-weight (2D, unit-batch) case only. MoE-style experts are a separate lever.

Out of scope: the MLP LX-overflow crash

This fix is about array-fill on the matmul. The separate prefill-MLP issue — the on-chip LX-residency / overflow behavior governed by the DeepTools LX allocator (a compile-time crash, not a utilization number) — is a different problem and is not touched here.

One-line reference: the attention fix

A separate commit (824ad4b on the sibling branch pr-mlp-fix) bypasses the SDPA fake-tensor setup that mis-fires on the Spyre device. It is an attention correctness fix, unrelated to this matmul layout work, and is mentioned only so it isn't confused with it.
