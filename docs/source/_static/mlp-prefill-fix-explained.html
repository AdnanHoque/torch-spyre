The MLP Prefill Fix: Rebalancing the Weight-Stationary Core Grid

Same 32 cores. Same math. Same K. The old schedule gave each core 64 M rows × 50 N sticks; the new one gives 128 × 25. Why is the second grid so much better? The answer is about weight duplication — and it has almost nothing to do with the number 25.

Grid: 8×4 (M-heavy) → 4×8 (N-heavy) Per core: 64×50 → 128×25 Weight duplication: 8× → 4× M-reuse / weight tile: 64 → 128
The puzzle this document exists to answer

The op is a shared-weight prefill projection: O[mb,out] = Σin A[mb,in] · W[in,out], with M = 512, N = 12800, K = 4096, on 32 cores.

Both the old and the new schedule describe the identical legal matmul. K is never split either way. So the FLOPs are identical, the result is bit-for-bit the same, and exactly 32 cores are busy in both. Yet the rebalance from an 8×4 grid to a 4×8 grid is a large win. How can rearranging the same 32 cores over the same matrix matter?

The answer rests on one asymmetry of a weight-stationary array: splitting M duplicates the weight; splitting N does not. This document builds that first principle from zero, then reads it straight out of the saved sdsc_0.json / jamie_unit_bmm_probe.md.

1Weight-stationary in one minute

Everything below falls out of one fact about how a Spyre core runs a matmul: it is a weight-stationary systolic array. To understand the fix you only need to internalize what that phrase costs you.

Each core has an 8×8 MPE array (the KG3 weight-stationary engine, 64 multiply-accumulate cells). To run a matmul tile, the hardware does three things, in order:

  • Load a tile of the weight W into the array and hold it resident. The weight values sit in the cells and do not move while the tile runs. This load is a fixed setup cost.
  • Stream the rows of the activation — the M dimension — through the resident weights, one after another. Each M row is a K-vector fed across the held weights.
  • The N outputs accumulate on the output edge as each M row passes.

The load-bearing consequence is amortization. Loading a weight tile is paid per weight load. You earn it back only by streaming many M rows past that resident tile before reloading. So a weight-stationary array has a clear preference, and it is the whole story of this fix:

★ The single rule

Load a weight tile once, then stream as many M rows as possible through it before you reload. Anything that forces you to load the same weight tile again — on another core, or another pass — is wasted setup the array did not need to pay.

2The asymmetry: M-split duplicates the weight, N-split does not

We have 32 cores and one big matmul. We must hand each core a rectangle of work. The work is indexed by two output axes — mb (the M rows of the output) and out (the N columns). So the scheduler chooses a grid: split M into m partitions and N into n partitions, with m × n = 32. Every core owns one cell (mb_i, out_j).

Here is the asymmetry that makes one grid better than another. Look at what a single core at cell (mb_i, out_j) actually needs to read:

The core at cell (mb_i, out_j) reads…which depends on…
its activation rows A[its M-rows, all in]its M-partition mb_i only
its weight columns W[all in, its out_j]its N-partition out_j only

Now ask: which cores share the same weight columns? Every core with the same out_j — regardless of its mb_i — needs the identical weight slice W[:, out_j]. They differ only in which M rows they stream through it.

Splitting M is expensive — it duplicates the weight

All the cores in one column of the grid (same out_j, different mb_i) need the same weight columns. So that weight slice is loaded once per M-partition. The whole weight W ends up loaded m times across the machine — the M-split factor is the duplication factor.

Splitting N is cheap — no duplication

Each N-partition owns different output columns and therefore different weight columns. Spreading more cores across N hands each one a fresh, non-overlapping slice of W. Parallelism goes up; weight loads do not. Every core still runs, but no weight byte is loaded twice.

✓ The first principle in one sentence

On a weight-stationary array, the M-split factor is the weight-duplication factor. Cores that share an N-partition reload the same weights; cores in different N-partitions never do. So given a fixed core budget, you want to spend cores on N (free parallelism) and as few as possible on M (each one re-loads the whole weight again).

3The money diagram: 8× vs 4× weight duplication

This is the diagram the entire fix turns on. The weight matrix W[in, out] is sliced by out into column-slices — one per N-partition. The height of each stack below is how many cores load that same slice, i.e. the M-split factor. That stack height is the number of times the weight is duplicated.

Why "load W 8×" is the right way to count it

There is exactly one copy of W in memory. "Loaded 8×" means the array setup — reading a weight slice into the resident MPE cells — happens 8 times for each slice's worth of weight across the 32 cores, because 8 cores in a column all need the same slice. That repeated setup is the cost a weight-stationary array is built to avoid.

⚠ The framing that is easy to get wrong

M was not absent or "demoted" before. The old SDSC clearly carried mb:512 and clearly split M — eight ways. The fix is not "introducing M streaming." It is two distinct things: (1) an identity change — the x:1 unit-BMM axis that lets DeepTools recognize this as the sendnn-like BMM and schedule it as such; and (2) a grid rebalance — moving cores off the M-split and onto the N-split (mb 8→4, out 4→8). Read the fix as rebalancing the 32-core grid, not as adding a stream that was missing.

5Rebalancing the 32-core grid: 8×4 → 4×8

Same 32 cells, both times. The rebalance just changes the shape of the grid: tall and narrow (M-heavy) becomes short and wide (N-heavy). Each cell's label is what that core computes.

6Reuse view: 64 vs 128 M-rows per resident weight tile

The duplication story has a mirror image that is worth seeing on its own, because it is the positive side of the same coin. A weight-stationary array wants to load a weight tile once and then stream as many M rows as possible through it. With fewer M-partitions, each core's resident weight tile serves more M rows before any reload.

7Why 25 beats 50 — the full accounting

Here is the most important caveat in the whole document: 25 N sticks per core is not magically better than 50 by itself. If you somehow shrank each core to 25 N sticks while keeping the rest of the grid the same, you would not gain anything. 25 wins only because of where the 25 came from.

★ The provenance of the 25

The 50→25 did not appear out of thin air. It is the direct consequence of moving cores from the M-split to the N-split (mb 8→4, out 4→8). More N-partitions ⇒ each owns fewer N sticks ⇒ 50 becomes 25. The 25 is a symptom of the rebalance, not the cause of the win.

So when we say "25 beats 50," what actually got better is everything the rebalance bought, all at once:

  • Fewer M partitions. mb 8 → 4. The M-split factor dropped by half.
  • Less duplicate weight loading. Because the M-split factor is the duplication factor, the full weight goes from being loaded 8× to 4× across the machine (Figure 1).
  • More M rows streamed per resident weight tile. 64 → 128 — double the weight-stationary amortization (Figure 3).
  • An N width that matches the backend's natural tile. 25 sticks is exactly one clean N tile (next section), where 50 is two of them.
✓ Net summary — state it plainly

25 N sticks/core is better here because it means fewer M partitions, less duplicate weight loading, more M rows streamed per resident weight tile, and an N width matching the backend's natural tile — all four are the same rebalance, seen from four angles. The number 25 is just the fingerprint of having spent the cores on N instead of M.

8The 25 matches the backend's natural N tile

There is one more reason the new grid lands cleanly, and it ties the work-division choice to the backend's own tiling. The full N dimension is 12800 / 64 = 200 output sticks. The KB's MLP-prefill model has an ideal N_TILE = 25 sticks, giving N_ITERS = 200 / 25 = 8 tiles over the whole N axis.

Putting the N-tile fact in its place

This is a real but secondary benefit. The dominant lever is the weight-duplication / reuse rebalance from sections 2–7. The N-tile match is the cherry on top: because the rebalance already chose 8 N-partitions, each core's 25 sticks happen to coincide with the backend's natural N_TILE = 25 — one iteration, no awkward remainder.

9Why correctness survived the change

A natural worry: if the schedule changed this much, how do we know the answer did not? Because both versions describe the same legal matmul, and every change was a scheduling/layout change, not an arithmetic one. Walk the four guarantees:

Same math

Both versions compute O[mb,out] = Σin A[mb,in]·W[in,out]. Identical contraction, identical operands.

Stick axes correct both times

Activation sticks on in (the K axis); weight and output sticks on out (the N axis). The fix did not move a stick axis, so the element-to-stick mapping is unchanged.

K is never split

in-split = 1 in both. With no cross-core reduction, there is no partial-sum accumulation across cores to get wrong — no reduction ambiguity at all.

Grids cover disjoint rectangles

8×4 and 4×8 both tile the output mb × out plane into 32 disjoint rectangles. Same area, repartitioned. No cell is computed twice or skipped.

✓ Dropping x:1 cannot change a value

The unit-BMM axis x has extent 1 — its only valid coordinate is zero. Adding or removing it cannot change any index or any sum; it only changes what scheduling and layout facts DeepTools can see (namely, that this is the sendnn-like BMM worth scheduling N-heavy). So the old version was not "wrong indexing." It was a correct but less favorable schedule: 64 rows × 50 sticks instead of 128 × 25, with no explicit unit-BMM identity to steer the planner.
