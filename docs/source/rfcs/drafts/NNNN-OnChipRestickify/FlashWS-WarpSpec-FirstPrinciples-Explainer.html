0The one-paragraph truth

Codex's "warp-specialized FlashAttention" is not a GPU warp-group kernel. It is a compiler-generated, loader-specialized K/V prefetch sidecar on top of the existing on-chip SDPA flash-prefill graph.

One core (core 31) takes a "loader" role: it issues a direct HBM→LX load of the next K/V tile while the other 31 cores compute the current attention tile, then fans that tile out over the ring into every core's scratchpad so the next matmul reads K/V from LX instead of re-streaming HBM. It is real, value-correct on standard inputs, and honestly fail-closed — but on the verified shape island it only ties the strong onchip_master baseline (geomean 0.99×); the "1.15×" headline is against a weak baseline codex itself later disowns. The genuine warp-group overlap was tried and failed value-correctness (NaN, 99% mismatch) and was abandoned for the conservative serialized loader.

Bottom line. A real, numerically-sound, exploratory probe — not a shippable performance win — and crucially it does not inherit the cross-shard reshard bug that broke our sibling attention-overlap work (§7).

1First principles: why K/V streaming is the bottleneck

FlashAttention removes the O(L²) score matrix from HBM. Once that's gone, the next pressure point is the repeated K/V stream — and on Spyre that stream hits the one slow shared pipe.

Every Q tile must re-see all of K/V, layout-converted for the QKᵀ and PV batchmatmuls. The traffic is ≈ 2·L·D·bytes·(L/Bq) — at L=1024, D=64, Bq=64 the K/V stream is re-read across 16 Q tiles (Stage094:152-178). The hardware asymmetry is what makes this the lever:

Figure 1. The asymmetry that defines the problem. HBM is one shared ~100 GB/s pipe; LX and the ring are ~45× faster and mostly idle during attention. FlashAttention already removed the O(L²) score traffic; what's left is the K/V stream re-read once per Q tile, all of it through HBM. Source: Stage094:38-41, 152-183.

2What "warp specialization" means here

On a GPU, warp specialization = different warp groups in one kernel issue to different units concurrently. The Spyre version is an analogue — codex is repeatedly explicit about that.

Figure 2. The GPU→Spyre mapping (Stage089:331-359). Tellingly, the string warpspec appears nowhere in onchip_realize.py or bundle.py — it is a config-preset name (§3, §6), not a code path.

3The mechanism — what the compiled bundle is

The standard ~11-SDSC SDPA graph, with the K/V-producer SDSC rewritten in place into a loader-specialized mixed SuperDSC, spliced into bundle.mlir — not a separate kernel.

Figure 3. The loader-core pipeline in time. Core 31 prefetches the future K/V tile (HBM→LX direct fill, onchip_realize.py:2808) while cores 0–30 compute; then a ring fanout makes the tile LX-resident for every core's next matmul. Schedule built by _kv_repack_hbm_prefetch_source_fanout_schedule (onchip_realize.py:4217).
Figure 4. Single-loader fanout topology (not an all-core HBM fill). Three storage identities coexist (Stage094:222-236): the preserved original HBM output (reductions still read it), the loader's LX source buffer, and the per-consumer LX input regions. Builder: build_flash_attention_kv_repack_hbm_prefetch_hoist_tile_artifacts (onchip_realize.py:4540).

Entry / dispatch: _apply_flash_attention_route_policy() (decompositions.py:86, called at :667) — this is the "core selector". The shared field-surgery primitive apply_lx_flip (onchip_realize.py:4953) only redirects an LX base pointer; it does not re-partition a shard — the detail that makes it safe (§7).

4Lineage — warpspec is the last step of a chain

Not a separate invention — a specialization on top of the on-chip handoff machinery. The crucial jump: Stage008 proved a double-buffered schedule but only LX→LX. Warpspec finally crosses the HBM→LX boundary.

Figure 5. The lineage. Warpspec reuses the handoff binding (LX bases, mixed SDSC, preserve-original-HBM-output) wholesale, scoped to the K/V operand of flash prefill.

5Layout decoupling — the turning point (Stage088)

The single most important engineering decision on the branch — and the reason it avoided the trap our sibling branch fell into.

Two mechanisms were originally coupled: a layout-transform pair (restickify/repack K/V) and the loader-core prefetch. Codex found that coupling them "hid valid warpspec rows behind layout-pair numerical failures" (Stage088:21-25).

Figure 6. Decoupling isolated what is being certified. The certified default is now the layout-free loader path. Codex diagnosed the risky reshard and walked away from it — on standard inputs — which is exactly why this branch does not carry our bug.

6What is actually proven

This is where the headline and the reality diverge. The verified ceiling ties the strong baseline; the real concurrency idea failed.

Figure 7. The 1.15× geomean is vs flash_hbm, which codex itself later relabels "not equivalent to upstream" (Stage098:200). Against the strong onchip_master the same run is 0.9929× — a net loss, 4 of 8 rows winning (Stage092:38-40). The decisive upstream three-way table was never produced — the clean build failed to compile (Stage098:234-250).

Maturity: a probe, not a product

  • Nothing is default-on — every gate defaults "0"/"-1" (config.py:55,308-489).
  • ~119 of 217 commits are docs-only stage notes.
  • The route policy is a hardcoded 4-shape lookup table (flash_attention_route_policy.py:31-38) — codex admits "no implemented production routing policy" (Stage094:757).
  • Causal is plan-artifact-only — causal_mask_dataop.py:456 self-reports requires_deeptools_dataop_parser_extension: True; never enters bundle.mlir.
  • The ~7k-line onchip_realize.py growth is ~30 diagnostic A/B knob paths — bring-up scaffolding.

7The question that matters for us: does it inherit our bug?

No — and this is the most important finding. The certified path uses a different, simpler, safer mechanism than the cross-shard reshard that broke our sibling attention-overlap branch.

Figure 8. The broken function does not exist on flash-ws (grep = 0 hits). The certified path is a same-layout fill + fanout, not a reshard. And codex independently hit the same 0.21–0.67 failure class in the layout-coupled variant (Stage086/087) and disabled the layout transform in response (§5) — diagnosing the risky reshard and walking away, on standard inputs, with a fail-closed gate.
Validation is rigorous. Standard torch.randn (no ×0.1), CPU F.scaled_dot_product_attention reference, max_abs_error ≤ 0.01 promotion gate (onchip_sdpa_sweep.py:1301-1328, onchip_sdpa_promotion_gate.py:520). Promoted rows report 0.002–0.006. This branch avoided the trap that bit us.

8Outlook — should we iterate?

Iterate for the mechanism, not for the current numbers.

DimensionAssessment
Mechanism real?Yes — a real mixed SuperDSC, executed on device, spliced into bundle.mlir
Numerically sound?Yes — standard randn, fail-closed gate, 0.002–0.006 error on certified rows
Inherits our cross-shard bug?No — different, simpler, safer mechanism (HBM→LX fill, no reshard)
Perf win over the STRONG baseline?No — geomean 0.99× vs onchip_master; the 1.15× is vs a weak baseline
The real overlap idea?Failed (NaN / 99% mismatch), abandoned for a serialized loader
CoverageNarrow — 8 rows, block64, non-causal; H8-long broken; causal plan-only
Productized?No — default-off, lookup-table "policy", scaffold-shaped code

What would have to become true for iteration to pay off

  1. Beat onchip_master, not flash_hbm — on a documented shape island. Today the serialized loader's own compute slice is a fixed cost only long rows amortize; the win is within noise.
  2. Recover real overlap safely — the same-row concurrency that failed (Stage052/053, read-after-write / no-producer) is the actual idea. The serialized-loader retreat is a correctness workaround, not the destination. High-value, high-risk frontier.
  3. A real route policy — replace the 4-shape frozenset with a cost model that predicts the crossover L (the first-principles arithmetic the docs narrate but never derive).
  4. Kernel-time measurement — all current numbers are host wall-time, tiny samples; re-measure on AIU kernel-time before trusting any speedup.
  5. Generalize past the island: batch>1 D128, H8-long, causal, block128 — each currently fails or is plan-only.
The recommendation in one line. The loader-core prefetch is a legitimately different and safer design than our asymmetric handoff — building on it does not inherit our bug, and codex's rigor (standard inputs, fail-closed gate, honestly-recorded failures) makes it a trustworthy foundation. But treat the perf as unproven: the certified result ties the strong baseline, the real concurrency idea is still unsolved, and the highest-value next step is exactly the thing codex could not make value-correct — genuine same-row producer/consumer overlap without the read-after-write hazard.
