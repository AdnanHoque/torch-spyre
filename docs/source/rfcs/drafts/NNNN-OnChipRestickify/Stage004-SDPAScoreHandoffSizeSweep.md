# Stage 004: SDPA Score Handoff Size Sweep

Date: 2026-05-24

## Purpose

This stage records the first multi-size latency comparison between stock
Torch-Spyre SDPA and the opt-in on-chip SDPA score-handoff realization.  The
question was whether the production-shaped Tier 1 primitive changes latency
across representative prefill sizes, and which pieces of the SDPA graph are
actually kept on chip.

## Method

All runs used the clean foundation DXP binary:

```sh
export HOME=/home/adnan-cdx
export DTI_PROJECT_ROOT=$HOME/dt-inductor-mixed
source "$DTI_PROJECT_ROOT/torch-spyre-docs/scripts/dev-env.sh"
export PYTHON="$DTI_PROJECT_ROOT/.venv/bin/python3"
export PATCHED_DXP="$DTI_PROJECT_ROOT/build/deeptools-onchip-foundation-clean/dxp/dxp_standalone"
export PATH="$(dirname "$PATCHED_DXP"):$PATH"
```

The benchmark shape was `(B=1, H=8, L, D=128)` with fresh
`TORCHINDUCTOR_CACHE_DIR` values for every run.  The baseline used stock SDPA:

```sh
SPYRE_FLASH_ATTENTION_PREFILL=0
SPYRE_ONCHIP_HANDOFF_REALIZE=0
SPYRE_ONCHIP_ATTENTION_SCORE_HANDOFF=0
```

The on-chip variant used the production flag set:

```sh
SPYRE_FLASH_ATTENTION_PREFILL=0
SPYRE_ONCHIP_HANDOFF_REALIZE=1
SPYRE_ONCHIP_ATTENTION_SCORE_HANDOFF=1
SPYRE_ONCHIP_HANDOFF_MIN_BYTES=1048576
```

Each timed sample measured the compiled Spyre call after warmup and called
`torch.spyre.synchronize()` before stopping the timer.  A CPU reference was
computed in FP32 and converted to FP16 for the max-error check.

## Descriptor Evidence

For realized sizes, the on-chip cache has two mixed score consumers:

```text
sdsc_4_max.json opFuncsUsed_=["STCDPOpLx", "STCDPOpLx"] datadscs_=2
sdsc_5_sub.json opFuncsUsed_=["STCDPOpLx", "STCDPOpLx"] datadscs_=2
```

The score producer is the first `batchmatmul` (`sdsc_3_batchmatmul`).  Its score
output is rewritten to LX-only.  For example, the `L=1024` descriptor contains:

```text
sdsc_3_batchmatmul Tensor2 OUTPUT:
  hbmStartAddress_=-1
  hbmSize_=0
  lxSize_=2147483647
```

`senprog.txt` was regenerated by copying each on-chip code directory and running
DXP with debug enabled:

```sh
DXP_DEBUG=1 "$PATCHED_DXP" --bundle -d /tmp/sdpa-sweep-onchip-L1024-debug-dxp
```

The realized score consumers show no HBM tokens and explicit L3 traffic:

```text
L=256  sdsc_4_max HBM=0 L3_LDU=64 L3_STU=64
L=256  sdsc_5_sub HBM=0 L3_LDU=64 L3_STU=64
L=512  sdsc_4_max HBM=0 L3_LDU=64 L3_STU=64
L=512  sdsc_5_sub HBM=0 L3_LDU=64 L3_STU=64
L=1024 sdsc_4_max HBM=0 L3_LDU=64 L3_STU=64
L=1024 sdsc_5_sub HBM=0 L3_LDU=64 L3_STU=64
```

The `L=2048` flagged run emitted no mixed SDSCs.  This is expected for the
current conservative Tier 1 realization: the per-core score slice is too large
for the reserved LX region model, so the implementation fails closed to the
stock path instead of emitting an uncertified program.

## What Stays On Chip

This is not full flash attention yet.  It is stock Inductor SDPA with one
specific tensor edge changed: the score matrix produced by `QK^T` is not written
to HBM for the realized sizes.  Instead, the score producer writes to LX and the
first two score consumers (`max` and `sub`, the front of softmax) are folded into
mixed SuperDSCs with `STCDPOpLx` data movement.

The rest of the graph remains the normal Torch-Spyre SDPA decomposition:

```text
mul -> mul -> ReStickifyOpHBM -> batchmatmul(QK^T score producer)
    -> max   [mixed, LX-fed]
    -> sub   [mixed, LX-fed]
    -> exp -> sum -> realdiv -> batchmatmul(PV) -> identity
```

So the architecture currently removes the highest-pressure score handoff from
HBM without replacing the whole SDPA algorithm.  The design intentionally keeps
the primitive narrow: same-stick `STCDPOpLx`, production descriptors, fail-closed
capacity checks, and no PT-LX restickify claim.

## Conclusion

The benchmark shows a real latency gain once the score handoff dominates enough
to amortize the mixed-SDSC data movement.  The clearest result is `L=1024`, where
median latency moved from `2.635 ms` to `1.924 ms` (`1.37x`).  Smaller sizes show
modest gains, and `L=2048` currently falls back because the simple LX reservation
model cannot host the required score regions.

The result supports continuing toward production on-chip SDPA, but the current
primitive should be described precisely as an on-chip score handoff for stock
SDPA, not as full flash attention/prefill fusion.
