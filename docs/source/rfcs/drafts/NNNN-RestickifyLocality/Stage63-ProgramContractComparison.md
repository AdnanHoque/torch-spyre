# Stage 63: Program Contract Comparison

## Summary

Stage 63 compared three related lowering paths for the LX-to-LX restickify
sidequest:

1. stock `ReStickifyOpHBM` generated by Torch-Spyre;
2. the DDL-shaped LX bridge from Stage 62;
3. standalone Deeptools data-op controls using `STCDPOpLx` and
   `ReStickifyOpLx`.

The result changes the direction of the prototype. The DDL bridge still gives an
HBM/L3-free generated artifact, but the patched template removes the explicit
producer-side LX read path. The data-op route gives a cleaner contract: when a
multi-data-op schedule is present, a two-step
`ReStickifyOpLx -> STCDPOpLx` Stage3B-shaped artifact lowers to Dataflow IR with
LXLU/LXSU work and no L3/HBM path.

This is still not hardware runtime proof, but it is stronger compiler evidence
for a viable LX-local restickify contract.

## Stock Versus DDL Bridge

The fixture is the clean Stage 62 case:

```python
def computed_transpose_adds_then_matmul(a, b, c, d):
    return (a + (b + c).t()) @ d
```

At size `2048`, stock Torch-Spyre emits one in-graph restickify inside the add
bundle:

```text
sdsc_fused_add_t_0:
  sdsc_0_add.json
  sdsc_1_ReStickifyOpHBM.json
  sdsc_2_add.json
```

The DDL bridge replaces only that restickify SDSC:

```text
sdsc_fused_add_t_0:
  sdsc_0_add.json
  sdsc_1_ReStickifyOpHBM_ddl_bridge.json
  sdsc_2_add.json
```

The top-level graph-free bundle and DSG files are nearly identical. The control
block shape does not obviously explain the timeout: the meaningful difference is
inside the generated restickify program.

Instruction-prefix summary for the restickify SDSC:

| path | retires on hardware | L3LU | L3SU | LXLU | LXSU | note |
| --- | --- | ---: | ---: | ---: | ---: | --- |
| stock `ReStickifyOpHBM` | yes | 128 | 128 | 64 | 64 | normal HBM/L3/LX lowering |
| DDL bridge | no | 0 | 0 | 0 | 32 | LX-store-side only in the summary |

The DDL bridge program contains many SFP/PT instructions, but it has no LXLU
unit programs. That is suspicious because a real LX-to-LX restickify needs both
a source-side read from producer-owned LX and a destination-side write to
consumer-owned LX.

## DDL Template Finding

The temp template used by the bridge is the Stage 49 passing spelling:

```ddl
%src_inp_lxsfp = ddl.unit(%inptensor) {unit="lxlu", data_connect="sfp_input"}
```

The stock Deeptools template uses an explicit external LX allocation:

```ddl
%inptensor_lx_allocation =
  ddl.get_external_data_transfer_allocation(%inptensor)
    {memory="lx", data_connect="lxlu_input"}

%src_inp_lxsfp =
  ddl.unit(%inptensor, %inptensor_lx_allocation)
    {unit="lxlu", data_connect="lxlu_input"}
```

Stage 49 showed that the explicit allocation form hits an LXLU/LRF address
boundary in some mirrored cases, while the no-allocation `sfp_input` form passes
DCC. Stage 63 explains the cost of that workaround: the generated Stage 62 DDL
bridge no longer has an obvious LXLU source program. It compiles, but it likely
lost the semantic link to the producer's actual LX allocation.

The bridge schedule also reflects this weak source contract:

```text
transfer_lds0_src:no_component_dst:lx_lx_local
...
transfer_lds1_src:lx_dst:no_component_lx_local
```

That is enough to get an LX-shaped program, but it is not enough evidence that
producer LX values are being read correctly.

## Data-Op Controls

The standalone data-op route gives a better reference contract.

Command shape:

```sh
SPYRE_RESTICKIFY_LX_DATAOP=1 \
python3 tools/restickify_lx_dataop_probe.py \
  --size 2048 \
  --num-cores 32 \
  --op STCDPOpLx \
  --stcdp-same-stick \
  --run-dcg \
  --mode baseline \
  --mode stage3b \
  --output-dir /tmp/stage63-dataop-compare/stcdp-control
```

`STCDPOpLx` lowers in both modes.

DCG log summary:

| artifact | DCG | DataOpStandalone | PCFG components |
| --- | ---: | ---: | --- |
| baseline `STCDPOpLx` | pass | pass | `L3SU : L3LU : LX : PE0` |
| Stage3B `STCDPOpLx` | pass | pass | `LX : PE0` |

Dataflow string-count summary:

| artifact | lines | L3LU hits | L3SU hits | LXLU hits | LXSU hits | send hits | HBM string hits |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| baseline `STCDPOpLx` | 41,307 | 1,604 | 1,856 | 224 | 192 | 320 | 128 |
| Stage3B `STCDPOpLx` | 1,447 | 0 | 0 | 224 | 192 | 64 | 0 |

The Stage3B same-owner control is the clean reference: LXLU/LXSU remain, while
L3LU/L3SU disappear.

## Two-Step Data-Op Result

The next test composed local restickification with same-stick movement:

```text
ReStickifyOpLx -> STCDPOpLx
```

Initially this artifact passed DCG but failed `DataOpStandalone` with:

```text
DtException: sdsc_.dataOpdscs_.size() == 1
/project_src/deeptools/dcc/src/Conversion/PCFGToDataflowIR/PCFGToDFManager.cpp line 89
```

The local Deeptools source explains why: `PCFGToDFManager` handles multiple
`datadscs_` only when `coreIdToDscSchedule` is populated. Without that schedule,
it falls into the single-data-op path and asserts `dataOpdscs_.size() == 1`.

Stage 63 updated the diagnostic `combine_dataop_sdscs()` helper to emit a
sequential per-core schedule for composed data-op payloads:

```json
"0": [[0, -1, 0, 1], [1, -1, 1, 0]]
```

That means:

```text
run data-op 0, sync after it;
then run data-op 1, sync before it.
```

With that schedule, the generated Stage3B two-step artifact lowers through
`DataOpStandalone`.

Dataflow summary:

| artifact | DataOpStandalone | lines | L3LU hits | L3SU hits | LXLU hits | LXSU hits | send hits | sync hits | HBM string hits |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| baseline two-step scheduled | pass | 42,680 | 1,347 | 1,600 | 512 | 448 | 640 | 1,280 | 128 |
| Stage3B two-step scheduled | pass | 4,040 | 0 | 0 | 512 | 448 | 256 | 640 | 0 |

This is the strongest compiler-side evidence so far for the data-op path:
Stage3B-shaped ownership continuity removes the L3 path while preserving
source-side and destination-side LX units.

## Code Change

The only code change in this stage is in the prototype data-op helper:

```text
torch_spyre/_inductor/codegen/restickify_lx_dataop.py
```

`combine_dataop_sdscs()` now fills `coreIdToDscSchedule` when more than one
`datadscs_` entry is composed. This keeps the generated two-step artifacts on
the Deeptools multi-data-op lowering path instead of the single-data-op fallback.

The focused unit test now checks that the schedule is emitted.

## Interpretation

The DDL bridge path is probably the wrong near-term proof vehicle. It gives an
attractive HBM-free program artifact, but the no-allocation template spelling
appears to remove the producer-side LX read contract. That matches the runtime
symptom: the bridge launches but does not retire.

The data-op path looks more principled:

- `ReStickifyOpLx` exists as a Deeptools data-op concept;
- `STCDPOpLx` proves same-stick LX movement;
- a scheduled `ReStickifyOpLx -> STCDPOpLx` sequence lowers to Dataflow IR;
- in the Stage3B-shaped case, the dataflow has no L3LU/L3SU/HBM string hits and
  keeps LXLU/LXSU work.

What remains missing is integration with the production bundle/runtime path.
Earlier evidence still applies: DXP rejects raw `datadscs_` in normal bundles
with "Datadsc not allowed, use dldsc". So this is not ready to replace
`ReStickifyOpHBM` in Torch-Spyre, but it gives us a cleaner lower-level target
than the patched DDL bridge.

## Recommendation

Stop pushing the current DDL bridge toward runtime until we can restore an
explicit producer-side LX read contract. The better next step is to pursue one
of these data-op integration routes:

1. find the current Deeptools DLDSc equivalent for the scheduled data-op sequence;
2. lower the scheduled two-step data-op through the Deeptools runtime pipeline
   that supports data-op DSCs, not the generic DXP bundle path;
3. if neither route exists, prototype the smallest DLDSc schedule that matches
   the Stage3B two-step dataflow.

For the ring-aware restickify project framing, this is a useful result: Stage3B
ownership continuity is exactly what lets the data-op Stage3B artifact remove
L3/ring-facing movement in the compiler IR.
