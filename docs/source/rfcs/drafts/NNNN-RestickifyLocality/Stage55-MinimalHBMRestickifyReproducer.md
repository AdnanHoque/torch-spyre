# Stage 55: Minimal HBM ReStickify Reproducer

## Summary

Stage 55 turned the Stage 54 launch-isolation result into a minimal reproducible
case for normal `ReStickifyOpHBM` execution. The reproducer is:

```python
def fn(a):
    return a.t().contiguous()
```

At size `128`, TorchInductor emits exactly one generated Spyre bundle containing
exactly one SDSC:

```text
sdsc_fused_clone_t_0
  sdsc_0_ReStickifyOpHBM.json
```

The bundle returns from `launch_kernel`, then hangs at the next
`RuntimeStream::synchronize`.

## Probe Change

`tools/restickify_scenario_probe.py` now includes two isolated restickify cases:

```text
isolated_transpose_contiguous
isolated_transpose_clone
```

These cases are intentionally small and are meant to reproduce restickify runtime
behavior without a fused add, matmul, or downstream consumer bundle.

## Command

```sh
timeout 120s python3 tools/restickify_scenario_probe.py \
  --case isolated_transpose_contiguous \
  --size 128 \
  --ring-telemetry \
  --skip-correctness \
  --sync-after-kernel \
  --kernel-launch-log \
  --output-dir /tmp/stage55-cli-isolated-contiguous-128 \
  --fail-on-error
```

## Result

The run timed out in:

```text
flex::RuntimeStream::synchronize
spyre::synchronizeDevice
```

The launch log shows:

```json
{"kernel_name":"sdsc_fused_clone_t_0","phase":"before_launch","sdsc_files":["sdsc_0_ReStickifyOpHBM.json"]}
{"kernel_name":"sdsc_fused_clone_t_0","phase":"after_launch","sdsc_files":["sdsc_0_ReStickifyOpHBM.json"]}
{"kernel_name":"sdsc_fused_clone_t_0","phase":"before_sync","sdsc_files":["sdsc_0_ReStickifyOpHBM.json"]}
```

There is no `after_sync` row.

## Generated SDSC Shape

The emitted SDSC is a clean HBM restickify:

```text
opFuncName: ReStickifyOpHBM
numCoresUsed_: 4
numWkSlicesPerDim_: mb:2, out:2
input layout:  mb, out   stick_dim=out
output layout: out, mb   stick_dim=mb
schedule nodes: allocate Tensor0_hbm, allocate Tensor1_hbm
segments: output=256, input=256, model=0, const=26
```

The segment sizes appear to be in 128-byte units, so `input=256` and
`output=256` correspond to the expected `128 * 128 * fp16 = 32768` bytes.

## Interpretation

This removes two possible explanations:

- The hang is not caused by the DDL bridge.
- The hang is not caused by fusion with add or matmul.

The smallest observed failing artifact is now a single normal
`ReStickifyOpHBM` SDSC generated by the current Torch-Spyre path.

The plain compiled add control still launches and synchronizes successfully, so
the pod/runtime is not generally unable to execute generated Spyre bundles.
