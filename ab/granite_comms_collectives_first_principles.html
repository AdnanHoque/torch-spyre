On-chip communication collectives for Granite

Turning LX relayout into a general comms substrate — the SDSC spill model, the collective taxonomy, and why the production contract is dl-dsc coordinates (#2939), not explicit movement (#2789).

When a producer op and its consumer prefer different work divisions over the same tensor, torch-spyre today spills the edge through HBM. The relayout work keeps that data resident in LX and moves it core-to-core over the L3 ring instead. The strategic question is who owns the physical movement: should Torch emit explicit transfer rows (#2789), or should Torch declare coordinates + a communication class and let Deeptools synthesize the movement (#2939)? This is the policy-vs-realization decision, and it decides whether the substrate scales from scatter to the full collective family.

The spill problem — from first principles

Each SDSC row is one lowered op. Four fields carry the relayout contract:

fieldmeaning
labeledDs_the op's tensors and whether each lives in HBM or LX.
scheduleTree_compute work division + tensor-distribution coordinates.
coreIdToWkSlice_which logical slice each core computes (the consumer's demand).
coordinates_.coreIdToWkSlice_which tensor slice already resides on each core for an LX input (the producer's residency).

An HBM spill arises when a producer builds a tensor under one core division and a consumer wants it under another. With no on-chip relayout the compiler materializes the edge through HBM — a ReStickifyOpHBM row, or HBM-backed consumer inputs. The production-shaped contract replaces that with a coordinate-mismatch pipeline:

producer tensor distribution  +  consumer compute distribution  →  coordinate mismatch  →  communication class  →  backend movement synthesis

The communication taxonomy

Every relayout edge is one of seven collective classes. Torch's job is to name the class (and cost it); the class tells the compiler the cardinality, whether arithmetic is involved, and how schedulable / risky the movement is. Below, each class drawn on cores — producers feeding consumers, with arrows = the physical ring movement Deeptools synthesizes. Only scatter is realized in PR1.

1. Scatter / permutation PR1 ✓

N→N, 1:1 ownership reassignment. No duplication, no arithmetic.

2. Broadcast not yet

1→ALL. One source piece copied to every consumer core. No arithmetic.

3. Multicast not yet

1→subset. Same as broadcast but only a cohort receives.

4. Gather not yet

MANY→1, fan-in of distinct pieces (assembly, no arithmetic). Not replicated.

5. All-gather / replicate unsafe full-resident

N→N, every core ends with EVERY piece. Attention value-side BMM operand.

6. Reduce needs reduce primitive

MANY→1 combined ARITHMETICALLY (+). Split-K partials. The "+" is what makes it not a copy.

7. All-reduce not yet

Reduce across all producers, THEN broadcast the result to all — reduce-scatter + all-gather. Combines the reduce-arithmetic and replication-sizing hazards.

The spine is identical in every class: Torch declares who owns what (producer→consumer coordinates) and the class label; the arrows — the physical ring movement — are what Deeptools synthesizes. Scatter is the one realized class; broadcast/multicast/gather are classified-but-unrealized; all-gather is realized-unsafe as full residency (needs loop-scoped fetch); reduce/all-reduce need a backend reduction primitive, not a copy.

Two architectures: who owns the movement?

Both PRs detect the same producer/consumer mismatch and keep data in LX. They differ on where the physical-movement detail lives.

Torch should own the physical movement intent, not the physical movement implementation.

Concretely, the #2939 frontend (torch_spyre/_inductor/lx_relayout.py) builds an LXRelayoutPlan per mismatched edge: it records producer_core_id_to_device_slice (the producer's residency, which becomes coordinates_.coreIdToWkSlice_ on the consumer's LX input) and a classification (scatter / layout_restickify_* / matmul_operand_broadcast), surfaced top-level as lxRelayoutClassifications_. It detects the edge purely by comparing producer vs consumer PerCoreView (if producer_view == consumer_view: continue), skips K-split partials, and emits no transfer rows. Deeptools reads that mismatch and synthesizes the movement.

The backend that makes scatter fit: deeptools PR #4408

PR #4408 (dxp/SdscRelayoutInsertion.cpp) is the synthesis half. The bug it fixes is a sizing error: the old code sized the inserted relayout as if the full tensor had to be materialized on every consumer core. For a Granite prefill attention tensor that overflows LX — the check fails and the edge falls back to an HBM restickify, even though the resident scatter piece is small.

Why #2939 is the production base

The distinction is not "coordinates versus transfers" — it is policy versus realization. The frontend must own the things only it knows: work-division selection, communication-class classification, the cost that feeds back into the planner, and whether an edge is worth keeping on chip. The backend should own the things that are hardware truths: ring movement, chunking/staging, legal STCDPOpLx/InputFetchNeighbor/ReStickifyOpLx lowering, IBUFF/LX/GTR constraints, and eventual compute/movement overlap.

#2789 is genuinely faster for forcing an isolated experiment — Torch can emit exactly the transfer it wants. But as the comm classes grow past resident scatter, that approach freezes backend and hardware constraints into Inductor: fanout schedules for broadcast, stage sizing for all-gather, stick-transform mechanics for restickify, arithmetic + synchronization for reduce. Policy scales with the number of communication classes; implementation scales with the number of hardware corner cases. Torch should carry the former.

The honest gaps. All-gather as full residency is unsafe today (IBUFF / bus-fence failures) — the fix is loop-scoped/tiled fetch, not bigger buffers. Gather is unvalidated end-to-end. Reduce/all-reduce need a backend reduction primitive that does not exist yet. These are classified and costed in the frontend (so the planner sees them) but deliberately not forced through resident scatter.

For Granite specifically, the next production-shaped work, in order: (1) keep weight restickifies out of scope; (2) implement layout_restickify_activation as an LX→LX layout transform; (3) implement the attention matmul_operand_broadcast as loop-scoped all-gather / input-neighbor movement (not full resident replication); (4) add gather / multicast / reduce / all-reduce only when a real transformer edge demands them — each costed in the frontend and synthesized in the backend.
