## Verdict

Model A (current behaviour): max peak LX/core = 2.00 MB on Llama 3.1 405B M=2048
Model B (expanded matmul):   max peak LX/core = 13.00 MB on Llama 3.1 405B M=2048

  Under expanded matmul-residency (B): peak 13.00 MB approaches/exceeds the cap.
  M4 has a real target IF the OP_OUTPUT_GOOD_FOR_LX_REUSE set is expanded. The current narrow set provides no target.

  VERDICT: M4 is conditionally viable. Sequence: first expand the eligible-op set (separate project), then reordering.

## Caveats

- Per-core peak under pure-M (32, 1, 1) divides total tensor by 32.
  Other splits (e.g., (1, 16, 2)) put more bytes on each core.
- Liveness model is approximate: tensors are assumed consumed by
  the very next op, missing residual-connection long-range liveness.
  This UNDER-estimates peak LX. Real peak could be ~2× this.
- Static analysis doesn't account for the LX allocator's
  fragmentation behaviour, which can effectively reduce usable bytes.
- Softmax intermediate handling is approximated as one resident
  output; the actual max/sum tensors are MUCH smaller. So Model A is
  an OVER-estimate of current peak LX.
